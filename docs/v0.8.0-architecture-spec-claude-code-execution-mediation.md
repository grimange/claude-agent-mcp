# claude-agent-mcp v0.8.0 Architecture Spec
## Claude Code Execution Mediation

## 1. Version

- Target version: `v0.8.0`
- Codename: `Claude Code execution mediation`

---

## 2. Objective

Advance Claude Code mode from structured continuity toward bounded execution realism.

v0.8.0 introduces a governed execution mediation layer that allows Claude Code backend runs to propose bounded follow-up actions which the runtime can validate, mediate, and execute under existing policy controls.

This is not native backend tool calling. It is runtime-mediated execution under explicit governance.

---

## 3. Why v0.8.0 exists

v0.6.0 expanded Claude Code capability in a bounded way.
v0.7.0 improved structured continuation and continuity observability.

The major remaining practical gap is that Claude Code mode still has:
- no native tool execution loop
- no backend-native persistent multi-turn execution model

v0.8.0 addresses this by introducing mediated execution requests while preserving honesty about backend limitations.

---

## 4. Design goals

### Primary goals
- Improve practical usefulness of Claude Code mode
- Allow bounded follow-up execution through governed mediation
- Preserve explicit policy enforcement and federation visibility rules
- Keep execution auditable and inspectable
- Avoid pretending the backend has native API-style tool calling

### Non-goals
- No native `tool_use` / `tool_result` protocol in Claude Code mode
- No streaming transport
- No cross-backend session migration
- No profile expansion release
- No breaking changes to top-level MCP tool contracts

---

## 5. Release theme

Claude Code may request bounded mediated actions; the runtime remains the enforcing authority.

---

## 6. Architectural direction

Add an internal execution mediation layer between Claude Code output and runtime action.

Flow:

1. Claude Code backend produces normal response text
2. Runtime detects a bounded mediated action request in a strict structured form
3. Runtime validates the request against policy, visibility, and allowed mediation rules
4. Runtime executes the allowed action
5. Runtime stores mediated execution results as session events
6. Results are summarized into continuation context for subsequent requests

This keeps the backend non-native while making the system more operational.

---

## 7. Core capabilities

## 7.1 Mediated action request model

Introduce a strict internal model such as `MediatedActionRequest`.

Suggested fields:
- `action_type`
- `target_tool`
- `arguments`
- `justification`
- `request_id`
- `mediation_version`

Supported action types in v0.8 should be minimal and bounded.

---

## 7.2 Allowed mediation scope

Only allow a narrow safe subset in v0.8.

Recommended initial scope:
- read-style tool requests
- bounded lookup requests
- non-destructive verification or inspection requests

Do not allow broad autonomous chaining or open-ended execution.

---

## 7.3 Runtime validation

Before execution, validate:
- target tool visibility
- federation allowlist compliance
- argument shape
- mediation scope eligibility
- action count limits
- request determinism constraints

Rejected requests must produce explicit operator-visible warnings and session events.

---

## 7.4 Mediated execution result model

Introduce a normalized `MediatedActionResult` model.

Suggested fields:
- `request_id`
- `status`
- `tool_name`
- `arguments_summary`
- `result_summary`
- `failure_reason`
- `policy_decision`

Persist results as session events and make them available to continuation reconstruction.

---

## 7.5 Execution mediation policy

Add bounded config for mediation behavior.

Suggested config:
- `claude_code_enable_execution_mediation`
- `claude_code_max_mediated_actions_per_turn`
- `claude_code_allowed_mediated_action_types`
- `claude_code_include_mediated_results_in_continuation`

Defaults should be conservative and disabled by default.

---

## 7.6 Continuation integration

When enabled, mediated action results should be summarized into continuation context.

Do not inject raw tool output indiscriminately.
Use compact summaries and bounded inclusion.

---

## 7.7 Capability contract v0.8

Add capability flags:
- `supports_execution_mediation`
- `supports_mediated_action_results`

Claude Code backend should only declare these true if the runtime integration is fully implemented and tested.

---

## 7.8 Observability

Add session events such as:
- `mediated_action_requested`
- `mediated_action_approved`
- `mediated_action_rejected`
- `mediated_action_completed`

Include policy decisions and bounded metadata for operator inspection.

---

## 8. API and contract impact

No breaking changes to:
- `agent_run_task`
- `agent_continue_session`
- `agent_get_session`
- `agent_list_sessions`
- `agent_verify_task`

Any mediation should remain an internal runtime/backend behavior.

---

## 9. Documentation requirements

Update:
- `docs/claude-code-backend.md`
- `docs/backend-capability-matrix.md`

Document:
- what execution mediation is
- what it is not
- policy constraints
- default disabled behavior
- operator-visible warnings and events
- preserved limitations

Be explicit that this is mediated execution, not native tool calling.

---

## 10. Testing requirements

Add tests for:
- mediated action request parsing
- validation and rejection paths
- allowlist and visibility enforcement
- result summarization
- continuation integration
- capability flags
- regression preservation
- deterministic mediation behavior for identical inputs

Full suite must pass.

---

## 11. Acceptance criteria

v0.8.0 is complete when:
1. Claude Code mode can request bounded mediated actions in a strict supported format
2. runtime validates and mediates eligible actions safely
3. rejected requests are explicit and observable
4. mediated results are normalized and persisted
5. continuation can summarize mediated results in a bounded way
6. external MCP contracts remain unchanged
7. docs describe the feature honestly
8. full suite passes

---

## 12. Preserved limitations

Still out of scope after v0.8.0:
- no native backend tool execution loop
- no true persistent backend-native multi-turn session
- no streaming transport
- no session migration
- no broad autonomous multi-step execution

---

## 13. Recommended implementation phases

### Phase 1 — models and config
- add mediation request/result models
- add config flags
- add capability flags

### Phase 2 — validation and policy
- implement mediated request parser
- implement validation against policy and visibility

### Phase 3 — execution integration
- execute approved bounded actions
- normalize and persist results

### Phase 4 — continuation integration
- summarize mediated results into continuation context

### Phase 5 — docs and tests
- add tests
- update docs
- run full validation

---

## 14. Final recommendation

Implement v0.8.0 as a bounded runtime mediation layer for Claude Code mode.

Do not overreach into native tool execution claims.

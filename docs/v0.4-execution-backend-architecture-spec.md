# claude-agent-mcp v0.4 — Execution Backend Track Architecture Specification

## 1. Purpose

This document defines the v0.4 **execution backend track** for `claude-agent-mcp`.

The purpose of v0.4 is to let operators choose between multiple execution backends while preserving the existing MCP surface, session model, policy model, artifact model, and canonical response contracts.

The motivating goal is to support both:

1. **API-backed execution**
   using Anthropic API credentials such as `ANTHROPIC_API_KEY`

2. **Claude Code-backed execution**
   using Claude Code authentication/session capabilities, which may be compatible with a Claude subscription in supported environments

v0.4 does **not** treat a Claude subscription as an API key.
Instead, it introduces a clean backend abstraction so the runtime can execute through either backend explicitly.

---

## 2. Baseline assumed from v0.3

The execution backend track builds on the existing baseline:

- stdio MCP transport
- Streamable HTTP transport
- canonical normalized response envelopes
- durable SQLite-backed sessions
- local filesystem artifact persistence
- built-in profiles:
  - `general`
  - `verification`
- existing tools:
  - `agent_run_task`
  - `agent_continue_session`
  - `agent_get_session`
  - `agent_list_sessions`
  - `agent_verify_task`
- hardened session locking and crash recovery
- local-first single-node deployment posture
- if v0.3 is complete, controlled downstream MCP federation

v0.4 must preserve these foundations.

---

## 3. Strategic objective

The execution backend track should answer this question:

**Can `claude-agent-mcp` preserve one stable governed runtime contract while allowing operators to choose how Claude execution is actually performed?**

Success means the answer is yes.

---

## 4. Execution-track goals

v0.4 exists to provide five capabilities:

1. **Explicit backend selection**
   Operators can configure which execution backend is used.

2. **Stable runtime contracts across backends**
   MCP tool contracts, session semantics, and response envelopes remain consistent.

3. **Backend-specific authentication handling**
   API mode and Claude Code mode authenticate differently, without pretending they are the same system.

4. **Backend capability normalization**
   Backend-specific differences are absorbed inside a backend adapter boundary.

5. **Future backend extensibility**
   The design can support additional backends later without changing the top-level runtime meaning.

---

## 5. Non-goals

The following are out of scope for v0.4 execution backend track:

- turning Claude subscriptions into API keys
- removing the existing API backend
- redesigning sessions around any backend-native session model
- adding new workflow tools
- adding new built-in profiles
- replacing SQLite
- replacing the canonical response envelope
- internet-scale hosting or multi-tenant execution
- broadening trust boundaries unrelated to backend selection
- rewriting the entire runtime around a backend-specific abstraction leaked upward

This is a backend selection release, not a product redefinition release.

---

## 6. Core principles

### 6.1 Internal contracts remain canonical

The source of truth remains the `claude-agent-mcp` internal session transcript, policies, and metadata.

### 6.2 Backend choice is explicit

Backend choice must be configured, not inferred magically from partial credentials.

### 6.3 Authentication models remain separate

API credentials and Claude Code authentication/session state are distinct and must not be conflated.

### 6.4 Capability differences are normalized internally

Different backends may support different capabilities, but those differences should be handled inside the execution layer.

### 6.5 Fail clearly on incompatible configuration

If a backend is selected without valid auth or runtime prerequisites, startup or execution should fail clearly and early.

---

## 7. Recommended backend model

v0.4 should introduce a formal execution backend interface.

Suggested internal abstraction:

- `ExecutionBackend`
- `BackendRegistry`
- `BackendSelectionResolver`
- `ApiExecutionBackend`
- `ClaudeCodeExecutionBackend`

Optional later:
- `AgentSdkExecutionBackend`

### 7.1 Minimum required backends

v0.4 should support at least:

- `api`
- `claude_code`

### 7.2 Optional later backend

If later useful, an Agent SDK backend can be added as a third backend, but it is not required for the first v0.4 slice.

---

## 8. Backend responsibilities

Each backend must implement a consistent internal contract for:

- backend identity
- backend availability checks
- startup validation
- task execution
- session continuation
- normalized output generation
- normalized error mapping

Suggested conceptual interface:

```python
class ExecutionBackend:
    name: str

    def validate_startup(self, config) -> None: ...
    def is_available(self, config) -> bool: ...
    def execute(self, request, session_state, visible_tools) -> BackendResult: ...
    def continue_session(self, request, session_state, visible_tools) -> BackendResult: ...
```

This is conceptual, not prescriptive exact code.

---

## 9. API backend

### 9.1 Purpose

The API backend preserves the existing programmatic path using Anthropic API credentials.

### 9.2 Authentication

The API backend is authenticated through API credentials such as:

- `ANTHROPIC_API_KEY`

### 9.3 Behavior

The API backend should continue to provide:

- bounded one-shot execution
- continuation based on internal transcript replay or current normalized behavior
- policy-bounded tool visibility
- normalized results and errors

### 9.4 Backward compatibility

The API backend should remain the backward-compatible default unless there is a strong product reason to change that default.

---

## 10. Claude Code backend

### 10.1 Purpose

The Claude Code backend allows execution through Claude Code rather than direct API calls.

This backend exists to support operators who want to use Claude Code authentication/session mechanisms, including supported subscription-backed usage patterns.

### 10.2 Authentication and environment model

The Claude Code backend must not use `ANTHROPIC_API_KEY` as a substitute for Claude Code login state.

Instead, it should rely on Claude Code’s supported runtime/auth environment.

The backend must clearly validate prerequisites such as:

- presence of Claude Code runtime/CLI if that is the chosen implementation path
- authentication availability in the local environment
- any required workspace/runtime assumptions

### 10.3 Implementation options

There are two possible implementation strategies:

1. **Claude Code CLI-backed backend**
2. **Claude Code library/backend path if officially supported and stable for this use case**

For the first v0.4 slice, the recommended practical choice is:

- **Claude Code CLI-backed backend**, if it is the most realistic route to subscription-compatible execution in the operator’s environment

### 10.4 Contract rule

Even if the backend uses Claude Code sessions or CLI semantics internally, `claude-agent-mcp` must keep its own canonical session model as the source of truth.

Backend-native session identifiers may be stored as optional metadata, not as the top-level session identity.

---

## 11. Backend selection and configuration

### 11.1 Required config

Add explicit backend selection to configuration.

Suggested values:

- `api`
- `claude_code`

Example conceptual setting:

- `CLAUDE_AGENT_MCP_EXECUTION_BACKEND=api`
- `CLAUDE_AGENT_MCP_EXECUTION_BACKEND=claude_code`

### 11.2 Backend-specific config groups

Configuration should support backend-specific settings.

Examples:

#### API backend
- API key location
- model selection if applicable

#### Claude Code backend
- CLI path if needed
- execution mode
- workspace assumptions
- timeout controls
- environment validation rules

### 11.3 Validation behavior

Startup validation should ensure:

- selected backend is known
- selected backend prerequisites are available
- incompatible mixed config is either ignored safely or rejected clearly

Do not silently fall back from one backend to another unless explicitly configured.

---

## 12. Runtime integration

### 12.1 Workflow executor

The workflow executor should remain the central orchestration layer.

Its responsibilities remain:

- resolve profile
- consult policy engine
- prepare session state
- resolve visible tools
- select execution backend
- invoke backend
- normalize result into canonical envelope
- update session and artifacts

### 12.2 Backend selection flow

Recommended flow:

1. config resolves selected backend
2. backend registry returns implementation
3. startup validation already confirmed availability
4. runtime execution calls selected backend
5. backend returns normalized internal result
6. workflow executor persists and formats result

### 12.3 No backend leakage to MCP layer

Public MCP tool responses should not expose backend-specific raw internals unless deliberately included in sanitized metadata.

---

## 13. Session model implications

### 13.1 Canonical source of truth unchanged

Internal session transcripts, summaries, and metadata remain canonical.

### 13.2 Optional backend metadata

A session may store backend metadata such as:

- `execution_backend`
- `backend_session_id` nullable
- `backend_run_id` nullable

These are optional accelerators or observability fields, not the canonical session identity.

### 13.3 Continuation semantics

Continuation must remain centered on the internal session model.
If the backend supports native continuation, it may be used as an optimization, but replay from internal state must remain possible where feasible.

---

## 14. Tool visibility and federation interplay

If v0.3 federation exists, visible downstream tools should continue to be resolved before backend execution.

Backends receive the filtered visible tool set.
They do not own federation policy.

This preserves:

- allowlist behavior
- profile-based visibility
- session audit logging

Backend choice must not bypass federation controls.

---

## 15. Error model

v0.4 may add backend-specific normalized errors such as:

- `ExecutionBackendConfigError`
- `ExecutionBackendUnavailableError`
- `ExecutionBackendAuthError`
- `ClaudeCodeUnavailableError`
- `ClaudeCodeInvocationError`

These errors should be normalized and should not leak raw backend internals unnecessarily.

---

## 16. Safety and operational posture

The execution backend track does not change the overall trust posture of the system.

The system remains:

- operator-controlled
- local-first
- single-node
- policy-bounded

However, docs must clearly explain:

- API backend billing/auth expectations
- Claude Code backend environment/auth expectations
- backend-specific limitations
- which backend is recommended for which operator goal

---

## 17. Documentation requirements

v0.4 should add or improve docs for:

- backend selection overview
- API mode setup
- Claude Code mode setup
- backend-specific prerequisites
- backend troubleshooting
- known behavioral differences, if any

Suggested docs:

- `docs/execution-backends.md`
- `docs/claude-code-backend.md`
- updates to `docs/deployment.md`
- updates to README

## 18. Testing strategy

Required test categories:

1. **Backend selection tests**
   - valid backend names accepted
   - invalid backend names rejected
   - default backend behavior explicit and tested

2. **Startup validation tests**
   - API backend missing API key fails clearly if required
   - Claude Code backend missing runtime/auth prerequisites fails clearly
   - mixed or conflicting config handled predictably

3. **Execution normalization tests**
   - run/continue through both backends produce normalized internal results
   - public canonical envelope remains stable

4. **Session metadata tests**
   - backend metadata recorded correctly when configured
   - canonical session identity remains unchanged

5. **Federation compatibility tests**
   - visible downstream tools remain bounded the same way across backends

Prefer test doubles for Claude Code execution where practical.
Do not rely on live subscription-backed environments in the default test suite.

---

## 19. Recommended implementation phases

### Phase 0 — Scope freeze

Confirm:
- execution backend track only
- no new workflows
- no session redesign
- no contract drift unrelated to backend selection

### Phase 1 — Backend abstraction and config

Add execution backend config, internal interfaces, and registry.

Checkpoint:
- API backend still works unchanged

### Phase 2 — API backend extraction

Refactor current execution logic into an explicit API backend implementation.

Checkpoint:
- no behavior drift for current API mode

### Phase 3 — Claude Code backend implementation

Implement the smallest practical Claude Code backend path.

Checkpoint:
- backend validates prerequisites clearly
- execution path can return normalized results

### Phase 4 — Session metadata and normalization

Record backend metadata in session state and normalize backend results/errors.

Checkpoint:
- MCP contracts unchanged
- backend differences contained internally

### Phase 5 — Docs and tests

Add backend docs, config docs, and tests.

Checkpoint:
- operator can intentionally choose `api` or `claude_code` with clear guidance

### Phase 6 — Stabilization

Run full validation and document any backend-specific limitations honestly.

---

## 20. Minimum viable v0.4 release definition

A release qualifies as v0.4 execution backend track only if all of the following are true:

- backend selection is explicit and validated
- API backend remains functional
- Claude Code backend is supported in a documented bounded way
- canonical MCP tool contracts remain stable
- internal session identity remains canonical
- backend-specific auth/config expectations are documented clearly
- tests cover backend selection and normalization behavior
- deployment and federation controls are not bypassed by backend choice

---

## 21. Recommended default

Recommended default backend:

- `api`

Reason:
- it preserves backward compatibility and clearer server-style behavior

The Claude Code backend should be opt-in and explicitly selected by operators who want that mode.

---

## 22. Summary

v0.4 execution backend track is the right step if the product goal is:

**support either API-key-backed execution or Claude-subscription-compatible Claude Code execution without changing the top-level MCP contract**

The correct design is not “subscription as API key.”
The correct design is:

- one stable governed runtime
- two explicit execution backends
- normalized results
- separate auth models
- preserved policy and session boundaries

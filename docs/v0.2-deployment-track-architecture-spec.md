# claude-agent-mcp v0.2 — Deployment Track Architecture Specification

## 1. Purpose

This document defines the v0.2 **deployment track** for `claude-agent-mcp`.

v0.2 deployment track does **not** expand the core product into downstream MCP federation or new workflow classes.
Its purpose is to make the existing v0.1 runtime easier to deploy, integrate, operate, and validate.

The deployment track builds on the hardened v0.1 core:

- canonical internal session model
- stdio MCP server
- SQLite-backed durable sessions
- local filesystem artifact persistence
- built-in profiles:
  - `general`
  - `verification`
- tool surface:
  - `agent_run_task`
  - `agent_continue_session`
  - `agent_get_session`
  - `agent_list_sessions`
  - `agent_verify_task`

v0.2 deployment track focuses on:

1. additional transport support
2. packaging and release ergonomics
3. operational clarity
4. integration smoke validation
5. deployment-safe configuration boundaries

---

## 2. Deployment-track goals

v0.2 deployment track exists to achieve five goals:

1. **Multiple supported transports**
   Add a network-capable MCP transport for host integration beyond stdio.

2. **Predictable packaging**
   Make installation, execution, versioning, and release behavior cleaner and more repeatable.

3. **Operator-ready configuration**
   Clarify environment variables, storage paths, timeouts, and deployment constraints.

4. **Deployment smoke confidence**
   Provide repeatable validation that the server starts and behaves correctly in supported deployment modes.

5. **No core contract drift**
   Preserve the v0.1 tool contracts, profiles, and session semantics while improving delivery.

---

## 3. Non-goals

The following remain out of scope for v0.2 deployment track:

- downstream MCP federation
- dynamic external tool imports
- new workflow tools
- new built-in profiles
- cancellation support
- public artifact browsing tools
- multi-tenant hosting
- distributed locking
- background job orchestration
- redesign of continuation semantics
- replacement of SQLite with a networked database

This is a deployment and operability release, not a capability-expansion release.

---

## 4. Strategic objective

The deployment track should answer this question:

**Can `claude-agent-mcp` be run and integrated reliably by more hosts and operators without changing what the core runtime means?**

Success means the answer is yes.

---

## 5. Scope of change

v0.2 deployment track should change only these areas:

### 5.1 Transport layer

Add one deployment-ready network transport.

Preferred priority order:

1. **Streamable HTTP**
2. **SSE**

Choose one as the primary v0.2 transport.
Do not overcommit to supporting both fully unless implementation is already straightforward and low-risk.

### 5.2 Packaging layer

Improve:

- package metadata
- CLI entry points
- version reporting
- install guidance
- release structure
- environment example clarity

### 5.3 Operational layer

Improve:

- startup behavior
- runtime configuration docs
- storage path management
- deployment troubleshooting
- smoke-test guidance

### 5.4 Test and validation layer

Add transport-aware smoke tests and packaging validation without introducing a heavy integration harness.

---

## 6. Core architectural rule

v0.2 deployment track must preserve:

- canonical response envelopes
- session semantics
- profile behavior
- policy enforcement
- provider adapter isolation
- current tool surface
- current persistence model

Transport changes must wrap the same runtime, not fork it.

---

## 7. Transport architecture

## 7.1 Required transport model

v0.2 must support:

- **stdio**
- **one deployment-grade network transport**

The server must expose the same tool contracts regardless of transport.

## 7.2 Preferred design

Implement a transport abstraction that routes all requests through a single server/runtime boundary.

Suggested pattern:

- `server.py` retains tool registration and common runtime wiring
- transport-specific bootstrap modules handle startup concerns
- tool handlers and workflow execution remain transport-agnostic

Suggested files:

```text
src/claude_agent_mcp/
  server.py
  transports/
    stdio.py
    streamable_http.py
```

If SSE is chosen instead of Streamable HTTP:

```text
src/claude_agent_mcp/
  transports/
    stdio.py
    sse.py
```

## 7.3 Transport selection

Transport selection should be explicit via configuration or CLI.

Example modes:

- `claude-agent-mcp --transport stdio`
- `claude-agent-mcp --transport streamable-http --host 127.0.0.1 --port 8000`

Do not auto-switch implicitly.

## 7.4 Transport safety constraints

Network transport in v0.2 must remain:

- local-first by default
- explicitly configured
- single-node
- bounded by the same policy engine
- non-multi-tenant

Default bind behavior should prefer loopback-safe local operation unless explicitly overridden.

## 8. Configuration model

v0.2 should formalize configuration categories.

## 8.1 Required config domains

- transport mode
- host
- port
- log level
- state directory
- SQLite path
- artifact directory
- default timeout caps
- default working-directory allowlist
- provider API key configuration

## 8.2 Configuration principles

- environment variables remain primary
- CLI overrides may be allowed for startup options
- configuration must be validated at startup
- invalid startup config must fail early and clearly
- operational defaults must be documented

## 8.3 Suggested environment variables

Examples:

- `CLAUDE_AGENT_MCP_TRANSPORT`
- `CLAUDE_AGENT_MCP_HOST`
- `CLAUDE_AGENT_MCP_PORT`
- `CLAUDE_AGENT_MCP_STATE_DIR`
- `CLAUDE_AGENT_MCP_DB_PATH`
- `CLAUDE_AGENT_MCP_ARTIFACT_DIR`
- `CLAUDE_AGENT_MCP_LOG_LEVEL`
- `ANTHROPIC_API_KEY`

These names may vary, but the configuration surface should be clean and explicit.

## 9. Packaging and CLI

## 9.1 Packaging goals

Packaging must support:

- editable local install
- reproducible normal install
- clear console entry point
- version visibility
- dependency grouping for dev/test

## 9.2 CLI goals

The main CLI should support:

- starting the server in stdio mode
- starting the server in network mode
- displaying version
- optionally running a lightweight health or self-check command if already compatible with the codebase

Avoid introducing a large CLI command tree if not necessary.

## 9.3 Versioning

v0.2 deployment track should formalize:

- package version location
- version exposure in package metadata
- version reporting in CLI and release docs

---

## 11. Smoke validation design

v0.2 should add small, trustworthy smoke validation rather than a giant integration suite.

## 11.1 Required smoke validations

At minimum validate:

- package installs
- CLI starts in stdio mode
- CLI starts in selected network mode
- MCP tools can be enumerated
- at least one `agent_run_task` path works through the supported transport stack under test or scripted validation
- persisted session state remains readable after restart in supported mode

## 11.2 Test philosophy

Prefer:

- small transport startup tests
- contract-level assertions
- lightweight scripted validation
- deterministic local-only test fixtures

Avoid:

- fragile end-to-end cloud-dependent tests
- giant framework-heavy orchestration

---

## 12. Security and deployment posture

v0.2 deployment track does not make the system multi-tenant or internet-safe by default.

It remains:

- operator-controlled
- local-first
- single-node
- policy-bounded
- not intended as an open public endpoint

Network transport must be documented with explicit caution around bind address, trust boundary, and authentication assumptions.

If authentication is not implemented, the docs must say so plainly.

## 14. Recommended implementation phases

### Phase 0 — Scope freeze

Confirm:

- deployment track only
- no federation
- no new workflow tools
- no contract drift

### Phase 1 — Transport abstraction

Refactor startup wiring so stdio and one network transport can share the same core server/runtime path.

Checkpoint:
- stdio still works unchanged

### Phase 2 — Network transport implementation

Implement one supported network transport.

Checkpoint:
- server starts cleanly
- tool enumeration works
- requests reach the same handlers as stdio

### Phase 3 — Configuration hardening

Implement or refine startup configuration validation.

Checkpoint:
- invalid startup config fails clearly
- docs reflect actual config behavior

### Phase 4 — Packaging and CLI polish

Improve package metadata, entry points, version exposure, and startup UX.

Checkpoint:
- install and startup are straightforward for operators

### Phase 5 — Docs and smoke validation

Add deployment docs and light smoke validation.

Checkpoint:
- a new operator can install, configure, start, and validate the server using docs alone

### Phase 6 — Release stabilization

Run full tests, smoke validation, and finalize release notes.

---

## 15. Minimum viable v0.2 deployment release definition

A release qualifies as v0.2 deployment track only if all of the following are true:

- stdio remains supported
- one deployment-grade network transport is supported
- public tool contracts are unchanged unless explicitly documented and backward compatible
- configuration is validated clearly at startup
- installation and startup docs are accurate
- package versioning is clean and visible
- smoke validation exists for startup and basic request flow
- the runtime remains single-node and policy-bounded
- no downstream federation or feature-scope creep was introduced

---

## 16. Recommended decision on transport

Preferred choice: **Streamable HTTP first**.

Reasoning:

- better deployment ergonomics
- cleaner host integration path
- simpler operator mental model in many environments

Fallback choice: **SSE** if it is significantly easier or better supported by the MCP SDK/runtime path you are using.

Do not choose both unless the second transport is nearly free once the first is implemented.

---

## 17. Summary

v0.2 deployment track is the right next step after post-hardening because it increases safe adoption without expanding the trust boundary.

The release should make `claude-agent-mcp`:

- easier to install
- easier to run
- easier to configure
- easier to validate
- easier to integrate

while preserving the meaning of the hardened v0.1 core.

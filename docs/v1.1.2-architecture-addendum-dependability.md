# claude-agent-mcp v1.1.2 Architecture Addendum
## Claude Code Availability Classification and Fail-Closed Unavailable Results

## 1. Objective

Improve the dependability of `agent_verify_task` in restricted verification corridors by ensuring Claude Code availability failures are detected, classified, and returned as machine-stable fail-closed outcomes.

This addendum introduces:

- distinct detection of Claude Code limit/unavailable conditions
- a machine-stable unavailable result contract
- short-circuit behavior to prevent pseudo-verification results
- explicit retry/fallback signaling for external orchestrators such as APNTalk/Codex

This is a dependability hardening patch.
It is not a provider-routing redesign.

---

## 2. Why this addendum exists

In restricted verification workflows, backend/provider availability failures must not be confused with actual verification outcomes.

The current risk is that Claude Code operational failures may appear as:
- vague text errors
- low-confidence or inconclusive verification outcomes
- generic not-verified results

That is unsafe for orchestration because APNTalk/Codex needs to distinguish:

- verification completed but failed
- verification completed but remained inconclusive
- verification never truly completed because Claude was unavailable or limit-reached

This addendum makes that distinction explicit and machine-readable.

---

## 3. Design goals

### Primary goals
- Detect Claude Code limit/unavailable conditions distinctly from normal verification failures
- Return machine-stable fail-closed unavailable results
- Prevent operational/backend failures from surfacing as pseudo-verification outcomes
- Expose deterministic retry/fallback signals for external routing logic

### Non-goals
- No automatic provider switching inside `claude-agent-mcp`
- No multi-provider abstraction redesign
- No transport redesign
- No change to admitted tool surface
- No weakening of restricted-mode fail-closed behavior

---

## 4. Release theme

When Claude Code is unavailable, `agent_verify_task` should return a clear operational-unavailable outcome rather than a misleading verification result.

---

## 5. Core capabilities

## 5.1 Separate operational failure taxonomy

### Goal
Separate backend/provider failures from verification-domain failures.

### Requirements
Add stable operational failure categories distinct from verification reason codes.

Suggested enums:

#### VerificationFailureClass
- `backend_unavailable`
- `backend_limit_reached`
- `backend_timeout`
- `backend_auth_failure`
- `backend_invocation_error`
- `backend_unusable_response`

#### VerificationFailureCode
Examples:
- `claude_code_limit_reached`
- `claude_code_not_installed`
- `claude_code_not_authenticated`
- `claude_code_timeout`
- `claude_code_process_error`
- `claude_code_empty_response`
- `claude_code_unparseable_response`

These must remain distinct from verification reason codes such as:
- `insufficient_evidence`
- `scope_too_broad`
- `out_of_profile_request`
- `restricted_mode_mismatch`

### Rationale
Operational failures are not verification results.

---

## 5.2 Top-level outcome distinction

### Goal
Give APNTalk/Codex a deterministic routing signal.

### Requirements
Add or extend a top-level verification outcome field such as:

- `verified`
- `not_verified`
- `inconclusive`
- `unavailable`

### Behavior
Use:
- `verified` when verification completed positively
- `not_verified` when verification completed and failed substantively
- `inconclusive` when verification completed but evidence/scope was insufficient
- `unavailable` when verification could not complete due to backend/provider availability issues

### Rationale
External orchestrators must not infer availability from prose.

---

## 5.3 Fail-closed unavailable result contract

### Goal
Return a stable machine-readable payload when Claude Code is unavailable.

### Requirements
When provider/backend failure is detected, `agent_verify_task` should return fields such as:

- `outcome_kind`
- `failure_class`
- `failure_code`
- `summary`
- `retryable`
- `fallback_recommended`
- `verification_performed`

Recommended semantics:
- `outcome_kind = unavailable`
- `verification_performed = false`

Optional:
- `fallback_class`
- `fallback_reason`

### Example shape
```json
{
  "outcome_kind": "unavailable",
  "failure_class": "backend_limit_reached",
  "failure_code": "claude_code_limit_reached",
  "summary": "Claude Code verification backend is currently unavailable due to usage limits.",
  "retryable": true,
  "fallback_recommended": true,
  "verification_performed": false
}
```

### Rationale
This gives APNTalk/Codex a safe fail-closed contract.

---

## 5.4 Claude Code failure normalization

### Goal
Convert raw subprocess/provider failures into stable outcome codes.

### Requirements
Add a normalization layer that maps operational signals into stable classifications.

Potential input signals:
- subprocess exit code
- timeout handling
- stderr output
- command-not-found
- authentication failure text
- usage/quota/limit text
- empty output
- malformed/unparseable output

Suggested internal helper:
- `classify_claude_code_failure(...)`

### Behavior
The classifier should produce stable:
- `failure_class`
- `failure_code`
- `retryable`

### Rationale
Raw CLI failures are too inconsistent for downstream orchestration.

---

## 5.5 Short-circuit before pseudo-verification

### Goal
Prevent backend unavailability from being mislabeled as a verification result.

### Requirements
If backend/provider failure is detected:
- do not continue into evidence assessment
- do not emit `inconclusive` merely because no evidence was produced
- do not emit `not_verified` unless substantive verification actually occurred

### Behavior
Operational failure should short-circuit directly to:
- `outcome_kind = unavailable`

### Rationale
This is the most important behavioral safeguard.

---

## 5.6 Retry/fallback signaling

### Goal
Support external orchestration without embedding full routing logic inside the server.

### Requirements
Expose explicit signals such as:
- `retryable`
- `fallback_recommended`

Optional:
- `fallback_class = codex`
- `fallback_reason = primary_verifier_unavailable`

### Behavior
These are advisory signals only.
They do not trigger automatic fallback inside `claude-agent-mcp`.

### Rationale
APNTalk/Codex can treat `claude-agent-mcp` as:
- primary verifier when available
- fallback trigger when unavailable

without forcing provider-switching into this release.

---

## 5.7 Restricted-mode compatibility

### Goal
Ensure the addendum works cleanly with APNTalk restricted verification mode.

### Requirements
In restricted mode:
- backend/provider unavailability remains fail-closed
- unavailable results remain explicit and machine-readable
- restricted-mode mismatch and provider-unavailable remain distinct conditions

### Behavior
For example:
- `restricted_mode_mismatch` is a policy/profile outcome
- `claude_code_limit_reached` is an operational availability outcome

These must not collapse into one category.

### Rationale
Safety and orchestration depend on this distinction.

---

## 6. API and contract impact

### Preserved behavior
- No new MCP tool required
- Existing restricted tool surface remains unchanged
- Existing verification-domain reason taxonomy remains valid

### Additive result behavior
`agent_verify_task` may return richer/additional fields:
- `outcome_kind`
- `failure_class`
- `failure_code`
- `retryable`
- `fallback_recommended`
- `verification_performed`

Where possible, these should be additive/defaulted for compatibility.

## 7. Data model impact

Prefer additive changes only.

- difference between verification outcomes and backend unavailability
- how unavailable results should be interpreted
- retry/fallback signaling semantics

## 9. Testing requirements

Add tests for:

### 9.1 Limit reached classification
- Claude Code limit/quota exhaustion maps to:
  - `outcome_kind = unavailable`
  - stable limit-reached failure code

### 9.2 Missing CLI classification
- missing `claude` binary maps to a stable unavailable code

### 9.3 Authentication failure classification
- expired/not-logged-in conditions map to a stable auth-failure code

### 9.4 Timeout classification
- timeout maps to:
  - unavailable
  - timeout-specific failure code

### 9.5 Empty/unusable response classification
- empty output
- malformed output
- unparseable output
map to stable unusable-response outcomes

### 9.6 No pseudo-verification leakage
- operational failures do not return:
  - `not_verified`
  - `inconclusive`
  - evidence-based reason codes

### 9.7 Retry/fallback signaling
- expected retryable/unretryable behavior is set correctly
- fallback recommendation appears where intended

### 9.8 Regression preservation
- normal successful verification behavior unchanged
- normal verification-domain failures unchanged
- restricted mode boundary behavior unchanged

---

## 10. Acceptance criteria

This addendum is complete when:

1. Claude Code unavailability and limit conditions are classified distinctly
2. `agent_verify_task` returns machine-stable unavailable results for backend/provider failures
3. operational failures never appear as normal verification outcomes
4. retry/fallback signaling is explicit and test-backed
5. normal verification behavior remains unchanged
6. restricted APNTalk-safe behavior remains intact

---

## 11. Preserved limitations

Still out of scope:
- no automatic Codex fallback execution inside the server
- no provider-routing engine
- no broad multi-provider abstraction redesign
- no widening of restricted verification surface

---

## 12. Recommended implementation phases

### Phase 1 — outcome and failure taxonomy
- add operational failure enums
- add unavailable top-level outcome

### Phase 2 — failure normalization
- classify Claude Code subprocess/provider failures into stable codes

### Phase 3 — short-circuit behavior
- prevent pseudo-verification on operational failure
- emit fail-closed unavailable results

### Phase 4 — docs and tests
- add unavailable-result docs/examples
- add classification and regression tests

---

## 13. Final recommendation

Implement v1.1.2 as a patch-level dependability hardening release.

The rule is:
- provider unavailable → unavailable result
- verification ran → verification outcome
- no mixing between the two

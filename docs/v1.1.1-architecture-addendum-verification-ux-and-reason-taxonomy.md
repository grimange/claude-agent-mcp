# claude-agent-mcp v1.1.1 Architecture Addendum
## Verification UX, Reason Taxonomy, and Preflight Guidance

## 1. Objective

Improve the operator usability of `agent_verify_task` by making low-signal, broad, incomplete, and out-of-profile requests easier to interpret.

This addendum introduces:

- a stable verification reason taxonomy
- a lightweight verification preflight/lint layer
- richer structured `agent_verify_task` results
- clearer operator guidance for narrow verification requests

These are quality-of-use improvements.
They must not weaken the existing safety boundary, restricted-mode behavior, or fail-closed guarantees.

---

## 2. Why this addendum exists

Restricted APNTalk-safe verification mode can be safe and machine-verifiable while still being frustrating to operate if results are difficult to interpret.

Current operator pain points include:
- broad requests producing low-signal outcomes
- insufficient evidence outcomes not clearly distinguishing cause
- profile mismatch not being clearly classified
- lack of concrete guidance on how to reframe a request

This addendum solves that by making verification outcomes more structured and operator-readable.

---

## 3. Design goals

### Primary goals
- Make `agent_verify_task` results easier to interpret programmatically and operationally
- Distinguish evidence problems from scope problems and profile/policy problems
- Catch obviously weak verification requests earlier
- Provide short actionable operator guidance
- Improve docs/examples for narrow verification requests

### Non-goals
- No new execution capability
- No widening of restricted tool surface
- No change to fail-closed boundary behavior
- No softening of restricted-mode rejection conditions
- No new public MCP tool required in the first implementation

---

## 4. Release theme

Verification results should be structured, interpretable, and operationally useful without changing the system’s safety posture.

---

## 5. Core capabilities

## 5.1 Stable verification reason taxonomy

### Goal
Introduce machine-readable reason codes for verification outcomes.

### Requirements
Add a stable enum, for example:

- `sufficient_evidence`
- `insufficient_evidence`
- `scope_too_broad`
- `out_of_profile_request`
- `restricted_mode_mismatch`
- `missing_required_context`
- `non_verifiable_request`
- `ambiguous_request`

Not all codes must be triggered in all releases, but the schema should support them.

### Behavior
`agent_verify_task` should return:
- `primary_reason`
- `reason_codes`

These should be stable codes, not only prose summaries.

### Rationale
This allows APNTalk and operators to distinguish:
- low evidence
- excessive scope
- profile mismatch
- outright non-verifiable requests

---

## 5.2 Verification preflight / lint pass

### Goal
Catch obviously weak verification requests before deeper verification work.

### Requirements
Add a lightweight internal preflight layer that evaluates request shape and profile alignment.

Suggested checks:
- overly broad scope
- missing target artifact or subject
- missing evidence anchor
- execution-oriented request in verification-only mode
- profile mismatch
- ambiguous verification goal

### Suggested model
A model such as:
- `VerificationPreflightResult`

Suggested fields:
- `ok`
- `lint_codes`
- `hints`
- `normalized_scope_summary`

### Behavior
In standard mode:
- preflight may produce warnings/hints and continue

In restricted APNTalk verification mode:
- strong mismatches may become direct verification blockers

Examples:
- execution-shaped request in verification-only mode
- request outside active profile
- request with no verifiable subject

### Rationale
This improves operator experience while preserving the safety boundary.

---

## 5.3 Richer structured verification result

### Goal
Make `agent_verify_task` results more interpretable without relying on prose alone.

### Requirements
Extend the result payload with structured fields such as:

- `decision`
- `primary_reason`
- `reason_codes`
- `summary`
- `operator_guidance`
- `evidence_sufficiency`
- `scope_assessment`
- `profile_alignment`

Suggested value families:

#### decision
- `verified`
- `not_verified`
- `inconclusive`

#### evidence_sufficiency
- `sufficient`
- `partial`
- `insufficient`

#### scope_assessment
- `narrow`
- `acceptable`
- `broad`
- `too_broad`

#### profile_alignment
- `in_profile`
- `out_of_profile`
- `restricted_mode_mismatch`

### Behavior
The result should distinguish between:
- a weak request
- a policy mismatch
- a failed verification
- an inconclusive verification

### Rationale
Operators need more than a yes/no style answer.

---

## 5.4 Distinguish request-quality failure from verification failure

### Goal
Avoid conflating poor input shape with negative verification findings.

### Requirements
Reason codes should map into at least three conceptual groups:

#### Request-quality reasons
- `scope_too_broad`
- `ambiguous_request`
- `missing_required_context`
- `non_verifiable_request`

#### Policy/profile reasons
- `out_of_profile_request`
- `restricted_mode_mismatch`

#### Evidence reasons
- `insufficient_evidence`

### Behavior
A request that is too broad should not be presented the same way as a request whose claim failed verification.

### Rationale
This is critical for operator trust and downstream automation.

---

## 5.5 Operator guidance layer

### Goal
Return short action-oriented hints alongside reason codes.

### Requirements
For each major reason code, provide one or more short guidance strings.

Examples:

#### insufficient_evidence
- "Provide the target artifact, expected outcome, or bounded evidence source."

#### scope_too_broad
- "Narrow the request to one artifact, one claim, or one verification objective."

#### out_of_profile_request
- "This request exceeds the active verification profile and should be handled outside restricted mode."

#### restricted_mode_mismatch
- "The active APNTalk verification mode only permits bounded advisory verification tasks."

### Behavior
Guidance should be concise, deterministic, and operator-oriented.

### Rationale
Reason codes alone are not enough for fast remediation.

---

## 5.6 Lightweight request-shape heuristics

### Goal
Improve handling of broad/incomplete prompts without introducing heavy NLP complexity.

### Requirements
Add a simple request-shape analysis model, for example:
- `VerificationRequestShape`

Suggested fields:
- `is_narrow`
- `breadth_score`
- `detected_targets`
- `detected_risks`

Heuristic signals may include:
- multiple asks in one request
- absence of named target artifact
- absence of expected outcome
- execution-oriented verbs in restricted verification mode
- vague “review everything” style language

### Behavior
This analysis may feed:
- preflight lint codes
- `scope_assessment`
- operator hints

### Rationale
A small deterministic heuristic layer is sufficient for this use case.

---

## 5.7 Integrated preflight inside agent_verify_task

### Goal
Improve verification UX without widening the public tool surface.

### Requirements
Integrate preflight directly into `agent_verify_task`.

Recommended flow:
1. parse request
2. run preflight/request-shape analysis
3. if restricted-mode hard mismatch is detected, return structured result immediately
4. otherwise continue verification
5. include preflight-derived findings in the final result

### Rationale
This preserves MCP tool stability while improving behavior.

---

## 5.8 Documentation and examples for narrow verification requests

### Goal
Reduce operator confusion before requests are even sent.

### Requirements
Add docs/examples showing:

#### Good narrow requests
- "Verify whether this specific file satisfies the APNTalk verification profile."
- "Check whether the runtime status proves restricted mode is active."
- "Verify whether the exposed tool list is exactly the admitted pair."

#### Bad broad requests
- "Review the whole system."
- "Tell me if everything is safe."
- "Validate the entire repo."
- "Check whether the whole integration is correct."

### Behavior
Docs should show likely reason-code outcomes for broad examples.

### Rationale
Examples often improve operator success more than internal logic alone.

---

## 6. API and contract impact

### Preserved behavior
- No new public MCP tool is required
- Existing restricted-mode boundary remains intact
- Existing tool-surface restrictions remain unchanged

### Additive result behavior
`agent_verify_task` may return richer structured result fields.
This should be additive where possible.

If a stricter schema change is required, preserve backward compatibility through optional/default fields.

## 9. Testing requirements

Add tests for:

### 9.1 Reason taxonomy
- reason codes returned consistently
- primary reason selection is deterministic

### 9.2 Broad request classification
- obvious broad requests return `scope_too_broad`

### 9.3 Insufficient evidence classification
- insufficient-evidence cases return `insufficient_evidence`

### 9.4 Profile mismatch classification
- out-of-profile cases return `out_of_profile_request`
- restricted-mode mismatch returns `restricted_mode_mismatch`

### 9.5 Operator guidance presence
- guidance strings present for major reason codes

### 9.6 Restricted-mode behavior preserved
- restricted-mode blockers remain blockers
- preflight does not widen permissions or soften fail-closed policy behavior

### 9.7 Backward compatibility
- standard mode and existing verification flows still work
- response compatibility preserved where intended

---

## 10. Acceptance criteria

This addendum is complete when:

1. `agent_verify_task` returns stable reason codes
2. broad/incomplete requests are classified more clearly
3. restricted-mode mismatch is explicit and machine-readable
4. operator guidance is returned for major negative/inconclusive outcomes
5. preflight is integrated without widening the public tool surface
6. docs/examples make narrow verification requests easier to author
7. existing safety and fail-closed behavior remain intact

---

## 11. Preserved limitations

Still out of scope:
- no new execution semantics
- no extra MCP tool required for linting
- no broad natural-language interpretation engine
- no weakening of restricted APNTalk mode
- no widening of admitted tool surface

---

## 12. Recommended implementation phases

### Phase 1 — reason taxonomy and result schema
- add reason enum
- add structured result fields
- map existing outcomes into stable codes

### Phase 2 — preflight and request-shape analysis
- add lightweight lint pass
- add request breadth/context heuristics
- integrate into `agent_verify_task`

### Phase 3 — operator guidance
- add deterministic guidance strings
- improve summaries for common outcomes

### Phase 4 — docs and tests
- add narrow-request docs/examples
- add classification and compatibility tests

---

## 13. Final recommendation

Implement this as a usability-focused extension to APNTalk-safe verification mode.

The rule is:
- clearer results
- better guidance
- same safety boundary

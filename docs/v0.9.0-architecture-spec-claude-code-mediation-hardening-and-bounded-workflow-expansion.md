# claude-agent-mcp v0.9.0 Architecture Spec
## Mediation Hardening and Bounded Workflow Expansion

## 1. Version

- Target version: `v0.9.0`
- Codename: `Mediation hardening`

---

## 2. Objective

Advance Claude Code execution mediation from single bounded action handling toward a more robust, configurable, and operator-safe mediation system.

v0.9.0 focuses on:
1. stronger mediation policy controls
2. richer rejection and audit diagnostics
3. bounded multi-action workflow support
4. improved continuation treatment for mediated results
5. preserved external MCP contracts and explicit safety boundaries

This release does not introduce native Claude Code tool calling, streaming transport, or cross-backend session migration.

---

## 3. Why v0.9.0 exists

v0.8.0 introduced:
- strict mediated action request parsing
- runtime policy validation
- approved execution through federation
- audit events
- optional continuation integration

That creates a usable foundation, but practical operator needs now shift to:
- tighter policy control
- clearer diagnostics
- slightly richer bounded workflows
- better mediated result handling across turns

v0.9.0 addresses those needs without widening into uncontrolled autonomy.

---

## 4. Design goals

### Primary goals
- Improve mediation safety and operator trust
- Expand from one-shot mediation to bounded workflow batches
- Make rejection reasons and policy decisions more explicit
- Improve auditability and continuation usefulness
- Preserve deterministic and governed execution behavior

### Non-goals
- No native `tool_use` / `tool_result` loop
- No streaming transport
- No cross-backend session migration
- No broad autonomous execution chains
- No breaking changes to top-level MCP tool contracts

---

## 5. Release theme

Claude Code may request a small bounded workflow, but the runtime remains the approving authority for every mediated step.

---

## 6. Architectural direction

v0.9.0 extends the mediation layer with a **bounded workflow model**.

Instead of only handling isolated action requests, the runtime may process a compact ordered set of mediated action requests under strict limits.

Each requested action remains individually validated and auditable.
No open-ended chaining is allowed.
No hidden loop execution is allowed.

---

## 7. Core capabilities

## 7.1 Bounded workflow request model

Introduce an internal workflow model such as:
- `MediatedWorkflowRequest`
- `MediatedWorkflowStep`
- `MediatedWorkflowResult`

A workflow is:
- ordered
- finite
- explicitly bounded by config
- individually validated per step

Recommended default maximum: 1 step unless explicitly increased.

---

## 7.2 Richer mediation policy profiles

Add mediation policy controls such as:
- allowed action types
- allowed tools
- denied tools
- maximum steps per turn
- maximum approvals per session
- continuation inclusion mode
- whether mixed action types are allowed in one workflow

These should remain conservative by default.

---

## 7.3 Explicit rejection diagnostics

Rejected mediated requests should provide normalized, inspectable reasons such as:
- feature_disabled
- invalid_version
- unsupported_action_type
- per_turn_limit_exceeded
- workflow_step_limit_exceeded
- federation_inactive
- tool_not_visible
- tool_not_allowed
- malformed_request

These reasons should appear in persisted events and normalized results.

---

## 7.4 Step-by-step auditability

Persist step-level events for workflow mediation, such as:
- `mediated_workflow_requested`
- `mediated_workflow_step_requested`
- `mediated_workflow_step_approved`
- `mediated_workflow_step_rejected`
- `mediated_workflow_step_completed`
- `mediated_workflow_completed`

All workflow handling must remain operator-inspectable.

---

## 7.5 Bounded continuation integration

Improve how mediated outcomes are summarized into continuation context.

Requirements:
- summarize only approved/completed steps by default
- support configurable inclusion of rejected-step summaries
- keep summaries compact and deterministic
- avoid raw output dumping

---

## 7.6 Capability contract expansion

Add capability flags such as:
- `supports_bounded_mediated_workflows`
- `supports_mediation_policy_profiles`

Only set these true when fully implemented and tested.

---

## 7.7 Operator-focused observability

Add concise stats and summaries for:
- requested workflows
- approved steps
- rejected steps
- omission/truncation of mediated summaries in continuation
- session-level mediation totals

This should improve trust and operational debugging.

---

## 8. API and contract impact

No breaking changes to:
- `agent_run_task`
- `agent_continue_session`
- `agent_get_session`
- `agent_list_sessions`
- `agent_verify_task`

Mediation workflow behavior remains internal runtime/backend behavior.

---

## 9. Data model changes

Suggested additive models:
- `MediatedWorkflowRequest`
- `MediatedWorkflowStep`
- `MediatedWorkflowResult`
- `MediationPolicyProfile`
- `MediationRejectionReason`

Prefer additive changes only.

---

## 10. Configuration changes

Add conservative config entries such as:
- `claude_code_max_mediated_workflow_steps`
- `claude_code_allowed_mediated_tools`
- `claude_code_denied_mediated_tools`
- `claude_code_max_session_mediated_approvals`
- `claude_code_include_rejected_mediation_in_continuation`
- `claude_code_mediation_policy_profile`

Defaults should preserve safe, low-expansion behavior.

---

## 11. Documentation requirements

Update:
- `docs/claude-code-backend.md`
- `docs/backend-capability-matrix.md`

Document:
- bounded workflow mediation
- policy profile behavior
- normalized rejection reasons
- continuation treatment of mediated results
- explicit preserved limitations

Be clear that this is still runtime-mediated execution, not native backend tool calling.

---

## 12. Testing requirements

Add tests for:
- workflow request parsing
- multi-step bounded approval/rejection behavior
- policy profile enforcement
- tool allow/deny controls
- rejection reason normalization
- continuation summarization rules
- capability flags
- regression preservation

Full suite must pass.

---

## 13. Acceptance criteria

v0.9.0 is complete when:
1. bounded mediated workflows are supported in a strict deterministic format
2. each workflow step is individually validated and auditable
3. rejection reasons are normalized and operator-visible
4. policy controls are richer and test-backed
5. continuation summaries remain bounded and useful
6. external MCP contracts remain unchanged
7. documentation is honest and complete
8. full test suite passes

---

## 14. Preserved limitations

Still out of scope after v0.9.0:
- no native tool calling in Claude Code mode
- no streaming transport
- no cross-backend session migration
- no broad autonomous or self-directed execution loops
- no backend-native persistent multi-turn session

---

## 15. Recommended implementation phases

### Phase 1 — models and policy surface
- add workflow and policy models
- add config and capability flags
- define rejection reason enums

### Phase 2 — parsing and validation
- implement workflow parsing
- validate steps individually
- enforce policy profiles and tool restrictions

### Phase 3 — execution and events
- execute approved bounded steps
- emit normalized workflow events
- build step and workflow result summaries

### Phase 4 — continuation integration
- integrate approved/rejected summaries into continuation context under bounded rules

### Phase 5 — docs and validation
- update docs
- add tests
- run full suite

---

## 16. Final recommendation

Implement v0.9.0 as a hardening and bounded workflow expansion release on top of the existing v0.8 mediation foundation.

Do not widen into transport, profile expansion, or native tool-calling claims.

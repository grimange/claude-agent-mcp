# claude-agent-mcp v1.0.0 Architecture Spec
## Stabilization, Operator UX, and Production Hardening

## 1. Version

- Target version: `v1.0.0`
- Codename: `Stabilization and hardening`

---

## 2. Objective

Deliver a production-ready v1.0.0 release of `claude-agent-mcp` by hardening the existing runtime, clarifying operator-facing behavior, simplifying configuration and policy understanding, and improving release confidence without widening scope into native tool calling or transport expansion.

v1.0.0 is a stabilization release.
It is not a speculative architecture pivot.

---

## 3. Why v1.0.0 exists

The project now has a coherent capability progression:

- v0.6.0: bounded Claude Code capability expansion
- v0.7.0: structured continuation and continuity observability
- v0.8.0: bounded execution mediation
- v0.9.0: mediation hardening and bounded workflow expansion

The remaining limitations are explicit and acceptable as product boundaries for a v1.0.0 release:
- no native `tool_use` / `tool_result`
- no streaming transport
- no cross-backend session migration
- no broad autonomous chaining
- mediated execution requires active federation

v1.0.0 should turn this into a clear, stable, documented product surface.

---

## 4. Design goals

### Primary goals
- Harden operator-facing behavior and error handling
- Improve configuration clarity and policy usability
- Improve documentation, examples, and upgrade guidance
- Improve auditability and runtime inspection ergonomics
- Preserve backward-compatible MCP contract stability
- Strengthen release/process confidence for production usage

### Non-goals
- No native backend tool-calling loop
- No streaming transport
- No cross-backend session migration
- No major new profile family
- No breaking changes to top-level MCP tool contracts

---

## 5. Release theme

v1.0.0 makes the current governed Claude Code execution model stable, understandable, and operationally ready.

---

## 6. Architectural direction

v1.0.0 focuses on stabilization layers around existing functionality rather than new capability families.

The release should improve:
- configuration ergonomics
- operator documentation
- policy inspection
- error normalization
- observability presentation
- release packaging confidence
- migration guidance across v0.6–v0.9

No new major execution primitive is required for v1.0.0.

---

## 7. Core capabilities

## 7.1 Configuration normalization and operator profiles

### Goal
Reduce operator confusion and make mediated execution policy easier to understand and manage.

### Requirements
Introduce a clearer operator-facing policy/profile layer that organizes current config into understandable presets or named modes.

Suggested additions:
- `safe_default`
- `continuity_optimized`
- `mediation_enabled`
- `workflow_limited`

These may map onto existing fields rather than replacing them.

### Behavior
- explicit precedence rules between profile presets and field overrides
- operator-inspectable resolved config/profile view
- no hidden behavior changes

### Rationale
The project now has enough flags that operators need a clearer mental model.

---

## 7.2 Resolved runtime status inspection

### Goal
Make it easy for operators to inspect what the runtime believes is enabled and supported.

### Requirements
Add a status/inspection surface that clearly reports:
- selected backend
- effective capability flags
- active mediation settings
- active continuation settings
- federation dependency status
- policy profile or resolved policy summary
- key preserved limitations

This may be exposed through an internal helper, CLI command, or docs-backed diagnostic output, depending on current architecture fit.

### Rationale
Operators should not need to infer effective behavior indirectly from logs.

---

## 7.3 Error and warning normalization polish

### Goal
Reduce ambiguity in operator-facing warnings and failure messages.

### Requirements
Review and standardize:
- downgrade warnings
- mediation rejection wording
- continuation truncation messaging
- federation-inactive messaging
- invalid or malformed mediated request messaging

### Behavior
- normalized phrasing
- stable categories/codes where appropriate
- no silent degradation for important policy/runtime conditions

### Rationale
A v1.0 system should be predictable under failure, not just under success.

---

## 7.4 Audit and observability presentation hardening

### Goal
Make audit trails easier to use in real operator workflows.

### Requirements
Add stable summarized views or helper rendering for:
- continuation reconstruction events
- mediation request/approval/rejection/completion events
- workflow summaries
- session-level totals and truncation indicators

Do not remove raw events; add clearer summarized presentation on top.

### Rationale
The runtime already emits rich signals. v1.0 should make them easier to operationalize.

---

## 7.5 Documentation and example coverage

### Goal
Make the supported model easy to adopt without guesswork.

### Required documentation improvements
- clarify supported vs unsupported behavior
- end-to-end examples for:
  - plain Claude Code execution
  - continuation sessions
  - single mediated action
  - bounded mediated workflow
  - policy allow/deny scenarios
  - federation inactive rejection behavior
- explicit operator setup guidance
- explicit upgrade notes from v0.6, v0.7, v0.8, and v0.9

### Rationale
This is essential for a credible v1.0 release.

---

## 7.6 Packaging and release hardening

### Goal
Increase confidence that packaged artifacts match tested behavior.

### Requirements
Review:
- version metadata consistency
- packaging contents
- docs inclusion
- release checklist quality
- install/test reproducibility
- optional smoke-test script or release validation checklist

### Rationale
v1.0 should be operationally releasable, not just test-passing in-repo.

---

## 7.7 Backward compatibility declaration

### Goal
State clearly what is stable in v1.0.

### Requirements
Document and preserve:
- top-level MCP tool contracts
- response envelope shape
- backward compatibility for v0.8 single-action mediation format
- backward-compatible defaults for additive capability/model fields

This should become an explicit compatibility statement in docs or release notes.

---

## 8. API and contract impact

No breaking changes to:
- `agent_run_task`
- `agent_continue_session`
- `agent_get_session`
- `agent_list_sessions`
- `agent_verify_task`

Any new inspection or helper surface should be additive.

---

## 9. Data model and config impact

Prefer additive changes only.

Potential additions:
- resolved policy/profile model
- operator status snapshot model
- normalized message category enums if useful

Do not redesign core runtime data flow for v1.0.

---

## 10. Documentation requirements

At minimum update:
- `docs/claude-code-backend.md`
- `docs/backend-capability-matrix.md`

Also add:
- `docs/operator-guide.md`
- `docs/upgrade-guide-v1.0.md`
- `docs/release-validation.md`

Documentation must clearly separate:
- what is supported
- what is intentionally unsupported
- what requires explicit opt-in
- what requires active federation

---

## 11. Testing requirements

Add tests for:
- resolved config/profile behavior
- status/inspection output
- normalized warning/error messaging where practical
- docs-linked example validity where testable
- regression preservation for v0.6–v0.9 behavior

Full suite must pass.

---

## 12. Acceptance criteria

v1.0.0 is complete when:
1. operator-facing configuration and policy behavior are clearer and test-backed
2. resolved runtime status/capability inspection is available
3. warnings and failure modes are more normalized and explicit
4. audit/observability presentation is easier to consume
5. operator docs and upgrade docs are complete
6. packaging/release validation guidance is present
7. top-level MCP contracts remain unchanged
8. full test suite passes

---

## 13. Preserved limitations

The following remain intentionally out of scope for v1.0.0:
- no native `tool_use` / `tool_result` loop
- no streaming transport
- no cross-backend session migration
- no broad autonomous execution chaining
- mediated execution still depends on active federation

These are product boundaries, not hidden deficiencies.

---

## 14. Recommended implementation phases

### Phase 1 — config and status clarity
- add resolved policy/profile inspection
- add status snapshot surface
- define precedence and display behavior

### Phase 2 — warning/error normalization
- standardize key warning and rejection presentation
- improve operator-visible messages

### Phase 3 — audit presentation
- add summarized session/audit inspection helpers
- improve continuity and mediation event readability

### Phase 4 — docs and examples
- add operator guide
- add upgrade guide
- add release validation checklist
- expand examples

### Phase 5 — validation and packaging
- run full regression suite
- verify packaging and release behavior
- finalize compatibility statement

---

## 15. Release note intent

v1.0.0 should be presented as:
- the first stable production-ready release of the governed Claude Code runtime model
- a stabilization and clarity release
- not a claim of native tool calling or transport parity with API-native systems

---

## 16. Final recommendation

Implement v1.0.0 as a stabilization, operator UX, and production-hardening release.

Do not dilute v1.0.0 with speculative transport or native-tool-calling work.

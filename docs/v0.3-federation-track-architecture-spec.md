# claude-agent-mcp v0.3 — Federation Track Architecture Specification

## 1. Purpose

This document defines the v0.3 **federation track** for `claude-agent-mcp`.

v0.3 federation track extends the deployable v0.2 runtime by allowing `claude-agent-mcp` to connect to selected downstream MCP servers and expose a controlled subset of downstream tools to Claude-backed sessions.

The purpose of v0.3 is **not** to become a generic open-ended tool router.
Its purpose is to add **governed downstream MCP interoperability** while preserving the existing core contract, policy model, and session semantics.

---

## 2. Baseline assumed from v0.2

The federation track builds on the existing v0.2 baseline:

- stdio MCP transport
- Streamable HTTP transport
- canonical normalized response envelopes
- durable SQLite-backed sessions
- local filesystem artifact persistence
- built-in profiles:
  - `general`
  - `verification`
- existing tools:
  - `agent_run_task`
  - `agent_continue_session`
  - `agent_get_session`
  - `agent_list_sessions`
  - `agent_verify_task`
- hardened session locking and crash recovery
- local-first single-node deployment posture

v0.3 must preserve these foundations.

---

## 3. Federation-track goals

v0.3 federation track exists to provide five capabilities:

1. **Static downstream MCP registration**
   Operators can configure one or more downstream MCP servers in local configuration.

2. **Startup-time downstream discovery**
   Downstream server tool discovery occurs at startup or explicit refresh time, not ad hoc during user execution.

3. **Explicit downstream tool allowlisting**
   Only a configured subset of downstream tools may be exposed.

4. **Profile-based downstream tool visibility**
   Profiles determine which classes or specific downstream tools may be visible to Claude sessions.

5. **Session-level auditability**
   Downstream tool availability and tool-call activity are recorded in session events for traceability.

---

## 4. Non-goals

The following are out of scope for v0.3 federation track:

- dynamic per-request downstream server registration
- automatic passthrough of all downstream tools
- unrestricted downstream tool exposure
- marketplace-style plugin loading
- remote untrusted tool discovery by default
- multi-tenant federation routing
- user-driven tool installation
- cancellation support
- new workflow tools
- public artifact browsing tools
- redesign of session semantics
- replacement of current persistence model

This is a **controlled federation** release, not a dynamic plugin ecosystem.

---

## 5. Strategic objective

The federation track should answer this question:

**Can `claude-agent-mcp` act as a governed agent hub that safely incorporates selected downstream MCP capabilities without losing policy control or contract stability?**

Success means the answer is yes.

---

## 6. Core federation principles

### 6.1 No ambient authority

Downstream tools must never become automatically available just because a server is reachable.

### 6.2 Startup-bound discovery

Tool discovery happens at startup or explicit admin refresh, not during arbitrary user requests.

### 6.3 Allowlist before visibility

Discovery alone is insufficient. A tool must be explicitly allowlisted before it can be surfaced to any Claude session.

### 6.4 Profile gating after allowlisting

Even an allowlisted tool is only visible if the active profile permits it.

### 6.5 Transport does not change trust

Federation is governed by policy and configuration, not by whether requests arrive over stdio or HTTP.

### 6.6 Canonical session model remains internal

Downstream tool calls are part of session events, but the source of truth remains the `claude-agent-mcp` internal session transcript and metadata.

---

## 7. High-level architecture

v0.3 federation introduces a bounded downstream tool federation layer.

Recommended internal components:

- `DownstreamRegistry`
- `DownstreamConnectionManager`
- `DiscoveredToolCatalog`
- `AllowedToolCatalog`
- `ToolVisibilityResolver`
- `DownstreamToolInvoker`

Suggested high-level flow:

1. server starts
2. configured downstream MCP servers are loaded
3. downstream servers are connected using configured transport settings
4. tools are discovered
5. discovery results are normalized into an internal catalog
6. allowlist policy is applied
7. profile visibility rules are applied at session execution time
8. selected downstream tools are injected into the provider runtime as callable tools
9. downstream tool call activity is logged in session events

## 9. Downstream server configuration model

## 9.1 Static registration

Operators register downstream MCP servers in configuration.

Each configured downstream server should include fields such as:

- `name`
- `transport`
- `command` or endpoint config
- `args`
- `env`
- `enabled`
- `discovery_timeout_seconds`
- `allowed_tools`
- `tool_name_prefix` or namespace mapping
- `profiles_allowed`

Example conceptual config:

```yaml
downstream_servers:
  - name: filesystem_tools
    transport: stdio
    command: python
    args: ["-m", "my_downstream_server"]
    enabled: true
    allowed_tools:
      - read_file
      - list_dir
    profiles_allowed:
      - general
```

## 9.2 Explicit enablement

A configured downstream server may exist but remain disabled.
Disabled servers are ignored at startup.

## 9.3 Safe defaults

Downstream federation should default to disabled unless explicitly configured.

---

## 10. Discovery and normalization

## 10.1 Discovery timing

Discovery happens at startup.
Optional explicit refresh may be added only if it remains operator-controlled and bounded.

## 10.2 Discovery output

Discovered downstream tools must be normalized into an internal catalog with fields such as:

- `downstream_server_name`
- `downstream_tool_name`
- `normalized_tool_name`
- `description`
- `input_schema`
- `enabled`
- `allowed`
- `profiles_allowed`

## 10.3 Name normalization

Downstream tool names should not be passed through raw if collisions are possible.
Each tool should have a stable normalized internal name.

Recommended pattern:

- `<server_name>__<tool_name>`

Example:

- `filesystem_tools__read_file`
- `search_tools__query`

This prevents collision and clarifies provenance.

## 10.4 Schema preservation

Input schemas from downstream tools should be preserved as faithfully as possible, but wrapped in internal metadata as needed for safe exposure.

---

## 11. Allowlisting model

## 11.1 Allowlist requirement

A discovered tool is not automatically usable.
It must be allowlisted in configuration.

## 11.2 Allowlist matching

Allowlisting may match:

- exact tool names
- explicit normalized names
- optionally simple exact server-scoped patterns

Avoid broad wildcard matching in v0.3 unless strictly necessary.

## 11.3 Deny by default

If a discovered tool is not allowlisted, it must be invisible and unusable.

## 11.4 No passthrough mode

There must be no “allow everything this server exposes” shortcut in v0.3.

---

## 12. Profile-based visibility

## 12.1 Visibility resolver

At session execution time, the active profile determines which allowlisted downstream tools are visible to the runtime.

## 12.2 v0.3 default posture

Recommended default:

- `general`: may use selected allowlisted downstream tools
- `verification`: downstream tools disabled by default

`verification` should only gain downstream visibility through explicit operator configuration and with careful safety review.

## 12.3 Visibility policy

Visibility should consider:

- profile name
- downstream server enabled state
- allowlist state
- profile allow rules
- runtime read-only or execution-class restrictions

---

## 13. Tool injection model

## 13.1 Runtime injection

Visible downstream tools are injected into the Claude-backed runtime as callable tools for that session.

## 13.2 Injection boundary

The provider adapter should receive only the filtered, visible tool set for the current session.
It should not know about all discovered tools globally.

## 13.3 Invocation path

Downstream tool invocation must flow through an internal invoker layer, not directly from provider output to external calls.

Responsibilities of the invoker:

- validate tool selection against visible set
- validate arguments against schema
- execute the downstream MCP call
- normalize success and error results
- record tool-call events in session history

---

## 14. Session and audit model

## 14.1 Session events

Session events should record:

- downstream tool catalog visible at execution start
- downstream tool selected
- invocation arguments summary
- normalized tool result summary
- invocation error summary if any

Sensitive content should be handled carefully and not overlogged.

## 14.2 Artifact implications

If downstream tool outputs are materialized as artifacts, those artifacts remain owned by `claude-agent-mcp` and should use the existing artifact model and provenance metadata.

## 14.3 Continuation semantics

Continuation must remain based on the canonical internal session transcript and metadata.
Downstream tool usage becomes part of the session’s event history.

---

## 15. Error model

Federation introduces new bounded error categories.

Suggested additions:

- `DownstreamServerConfigError`
- `DownstreamDiscoveryError`
- `DownstreamToolNotAllowedError`
- `DownstreamToolNotVisibleError`
- `DownstreamInvocationError`
- `DownstreamSchemaValidationError`

These errors should be normalized and should not leak raw downstream transport internals into public MCP contracts unless needed for debugging and safely sanitized.

---

## 16. Safety and security posture

v0.3 federation expands the trust boundary and must be treated accordingly.

Minimum rules:

- downstream federation is opt-in
- downstream servers must be explicitly configured
- tools are deny-by-default
- `verification` remains downstream-disabled by default
- HTTP exposure without authentication must not be mistaken for public-safe federation
- docs must state that federation is intended for trusted operator-controlled environments

If streamable-http remains unauthenticated, federation should still be documented as safe only in trusted deployments.

---

## 17. Configuration model additions

Suggested new configuration domains:

- downstream federation enabled flag
- downstream server definitions
- per-server discovery timeout
- per-server allowed tool list
- per-server profile visibility rules
- optional namespace prefix settings

These may live in file-based config, environment-backed structured config, or another explicit configuration mechanism already consistent with the repo.

Do not force complex dynamic config management in v0.3.

---

## 18. Testing strategy

Federation tests should remain bounded and deterministic.

Required test categories:

1. **Registry tests**
   - disabled servers ignored
   - configured servers loaded correctly
   - invalid configs fail early

2. **Discovery tests**
   - discovered tools normalized correctly
   - collisions handled via normalized names
   - non-allowlisted tools remain unavailable

3. **Visibility tests**
   - `general` sees only explicitly permitted tools
   - `verification` sees no downstream tools by default
   - profile overrides behave correctly

4. **Invocation tests**
   - allowed visible tools invoke successfully
   - invisible tools cannot be invoked
   - schema-invalid calls fail cleanly
   - downstream failures normalize cleanly

5. **Session audit tests**
   - downstream tool usage is recorded in session events
   - summaries are persisted without corrupting session state

Prefer test doubles for downstream servers where possible.
Avoid heavy real-network federation tests as the default suite.

---

## 19. Recommended implementation phases

### Phase 0 — Scope freeze

Confirm:
- federation track only
- no new workflow tools
- no cancellation
- no artifact browsing
- no contract drift unrelated to federation

### Phase 1 — Federation config and models

Implement downstream config schema, internal models, and startup validation.

Checkpoint:
- invalid downstream config fails early
- federation remains disabled by default

### Phase 2 — Discovery and catalog normalization

Implement startup discovery and normalization into an internal catalog.

Checkpoint:
- discovered tools are visible in internal state only
- no tool is usable without allowlisting

### Phase 3 — Allowlist and profile visibility

Implement allowlist filtering and profile-based visibility resolution.

Checkpoint:
- `general` can receive explicitly allowed tools
- `verification` remains empty by default

### Phase 4 — Invocation layer

Implement bounded downstream tool invocation and normalization.

Checkpoint:
- selected downstream tools can be called through the runtime
- failures are normalized

### Phase 5 — Session audit integration

Log visible tool sets and invocation summaries into session events.

Checkpoint:
- downstream usage is traceable in session history

### Phase 6 — Docs and stabilization

Add federation docs, policy guidance, and tests.

Checkpoint:
- operators can configure one downstream server safely using docs alone

---

## 20. Minimum viable v0.3 federation release definition

A release qualifies as v0.3 federation track only if all of the following are true:

- downstream federation is opt-in and disabled by default
- at least one downstream server can be statically registered
- startup discovery works
- discovered tools are normalized into an internal catalog
- only allowlisted tools are visible
- profile-based visibility is enforced
- `verification` remains downstream-disabled by default
- downstream invocation is normalized and audited in session events
- existing v0.2 tool contracts remain stable
- docs explain trust boundaries and limitations honestly

---

## 21. Recommended first federation slice

The best first slice for implementation is intentionally small:

- one downstream server
- startup-time discovery
- exact allowlisted tool names only
- normalized internal tool names
- visibility enabled only for `general`
- no dynamic refresh
- no wildcard allowlists
- no federation exposure for `verification`

This slice is enough to prove the architecture without opening the trust boundary too far.

---

## 22. Summary

v0.3 federation track is the right next step after v0.2 deployment track because the project is now deployable enough to support a controlled capability expansion.

The correct goal is not “more tools at any cost.”
The correct goal is **governed downstream interoperability**:

- static
- allowlisted
- profile-gated
- auditable
- bounded

# claude-agent-mcp v0.7.0 Architecture Spec
## Session Continuity and Execution Ergonomics

## 1. Version

- Target version: `v0.7.0`
- Codename: `Claude Code session continuity`

---

## 2. Objective

Advance the Claude Code backend from bounded capability expansion toward a more operationally coherent session model.

v0.7.0 focuses on:

1. stronger continuation semantics
2. structured session reconstruction
3. continuation-oriented execution ergonomics
4. explicit backend continuity contracts
5. preserved safety and downgrade honesty

This release does **not** attempt full native downstream tool execution, full streaming transport, or cross-backend session migration.

---

## 3. Why v0.7.0 exists

v0.6.0 introduced:
- limited downstream tool forwarding
- continuation prompt framing
- downgrade visibility
- cross-backend contract validation

However, the current Claude Code mode still has these limitations:
- no native multi-turn execution model
- history reconstructed as plain text
- each call is still one CLI subprocess
- no structured continuity model beyond prompt framing

v0.7.0 addresses those continuity gaps without pretending Claude Code mode has native API-grade session behavior.

---

## 4. Design goals

### 4.1 Primary goals
- Make continuation behavior more deterministic and inspectable
- Reduce prompt ambiguity during resumed sessions
- Introduce a structured session reconstruction model
- Preserve stable top-level MCP contracts
- Keep downgrade behavior explicit and operator-visible

### 4.2 Non-goals
- No real `tool_use` / `tool_result` loop in Claude Code mode
- No SSE or Streamable HTTP transport
- No cross-backend session migration
- No new planner/coder profile family in this release
- No breaking changes to top-level MCP tool contracts

---

## 5. Release theme

**Theme:** Claude Code continuity should become more structured, deterministic, and observable, while remaining honest about backend limits.

---

## 6. Architectural direction

v0.7.0 adds a **session continuity layer** for Claude Code execution.

Instead of treating continuation as only a prompt-header difference, the runtime will construct a more formal continuation package made from:

- session summary
- recent turn history
- prior warnings relevant to continuity
- prior forwarding decisions relevant to continuity
- active execution context metadata
- continuation directives

The Claude Code backend will still execute through subprocess invocation, but the input to that subprocess will become more structured and versioned.

---

## 7. Core capabilities

## 7.1 Structured session reconstruction

### Goal
Replace purely ad hoc history reconstruction with a deterministic, structured session reconstruction pipeline.

### Requirements
The runtime should construct a `SessionContinuationContext` object before calling the Claude Code backend.

Proposed fields:
- `session_id`
- `task_id`
- `is_continuation`
- `session_summary`
- `recent_user_requests`
- `recent_agent_outputs`
- `relevant_warnings`
- `forwarding_history`
- `active_constraints`
- `continuity_notes`
- `reconstruction_version`

### Behavior
- Continuation calls should use structured reconstruction inputs rather than raw freeform concatenation
- Reconstruction output should be deterministic for identical persisted session state
- Reconstruction should prefer bounded recent context plus stable summary over unbounded history inclusion

### Rationale
This improves consistency and operator trust while staying within the limits of a CLI backend.

---

## 7.2 Continuation window policy

### Goal
Define a formal policy for how much prior context is included in Claude Code continuation prompts.

### Requirements
Add configurable policies for:
- max recent turns included
- max warning entries included
- max forwarding events included
- whether to include verification outcomes
- whether to include prior dropped-tool warnings

### Proposed config fields
- `claude_code_max_continuation_turns`
- `claude_code_max_continuation_warnings`
- `claude_code_max_continuation_forwarding_events`
- `claude_code_include_verification_context`
- `claude_code_include_tool_downgrade_context`

### Defaults
Use conservative defaults that keep prompts compact and deterministic.

### Rationale
This keeps continuation usable without letting reconstructed history grow uncontrolled.

---

## 7.3 Explicit continuation package rendering

### Goal
Render continuation state into a stable, inspectable prompt structure.

### Required prompt sections
For continuation calls, the structured prompt should support sections such as:
- `[Continuation Session]`
- `[Session Summary]`
- `[Recent Interaction State]`
- `[Relevant Warnings]`
- `[Tool Forwarding Context]`
- `[Active Constraints]`
- `[Current Request]`
- `[Instructions]`

### Behavior
- Omit empty sections deterministically
- Preserve section ordering
- Ensure initial requests do not use continuation-only sections unless `is_continuation=True`

### Rationale
This preserves prompt determinism while improving continuity clarity.

---

## 7.4 Continuity-aware warnings model

### Goal
Prevent stale or irrelevant warnings from polluting continuation prompts.

### Requirements
Introduce warning classification for continuity relevance:
- `continuation_relevant`
- `operator_only`
- `request_local`

Only continuation-relevant warnings should be eligible for inclusion in continuation reconstruction by default.

### Behavior
- Tool downgrade warnings may be included when they affect current session capability interpretation
- One-off request-local warnings should not automatically carry forward
- All warnings remain in persisted records, but only a filtered subset is used for continuation prompts

### Rationale
This avoids noisy prompt carryover and improves session coherence.

---

## 7.5 Continuity-aware forwarding memory

### Goal
Carry forward only the forwarding information that matters for resumed work.

### Requirements
Persist forwarding decisions in a normalized form that can be summarized into:
- forwarding mode
- compatible forwarded tools
- dropped tools
- most recent drop reasons

### Behavior
On continuation:
- include forwarding context only when limited forwarding is enabled or when previous downgrade affects operator expectations
- keep formatting bounded and compact
- avoid repeating the full tool catalog

### Rationale
The model needs continuity of capability context, not raw repetition.

---

## 7.6 Backend capability contract v0.7

### Goal
Make continuity support part of the backend contract surface.

### Proposed `BackendCapabilities` additions
- `supports_structured_continuation_context: bool`
- `supports_continuation_window_policy: bool`

### Expected values
- Claude Code backend: `True`
- API backend: may be `True` if supported cleanly, otherwise `False` until implemented consistently

### Rationale
This makes continuity behavior explicit instead of implicit.

---

## 7.7 Session inspection support

### Goal
Make reconstructed continuity visible to operators for debugging and trust.

### Requirements
Persist session events describing:
- reconstruction policy applied
- counts of included turns/warnings/forwarding entries
- omitted context counts
- reconstruction version

### Behavior
Add session events such as:
- `session_continuation_context_built`
- `session_continuation_context_truncated`
- `session_continuation_prompt_rendered`

### Rationale
Operators should be able to inspect what continuity state was carried into a resumed Claude Code invocation.

## 8. API and contract impact

## 8.1 Top-level MCP tools
No breaking changes to:
- `agent_run_task`
- `agent_continue_session`
- `agent_get_session`
- `agent_list_sessions`
- `agent_verify_task`

## 8.2 Internal backend interfaces
Expected extension points:
- continuation context builder
- continuation window policy resolver
- structured prompt rendering helpers
- warning relevance classifier
- forwarding memory summarizer

---

## 9. Data model changes

## 9.1 New internal models
Suggested new models:
- `SessionContinuationContext`
- `ContinuationWindowPolicy`
- `ContinuationRenderStats`
- `ContinuationRelevantWarning`
- `ForwardingContinuationSummary`

## 9.2 Persistence expectations
No migration-heavy redesign required unless current persistence cannot support summary fields cleanly.

Prefer additive persistence changes only.

---

## 10. Configuration changes

Add bounded config entries for continuity behavior:

- `claude_code_max_continuation_turns: int`
- `claude_code_max_continuation_warnings: int`
- `claude_code_max_continuation_forwarding_events: int`
- `claude_code_include_verification_context: bool`
- `claude_code_include_tool_downgrade_context: bool`

All defaults should favor:
- deterministic behavior
- compact prompts
- backward compatibility
- explicit operator control

---

## 11. Documentation requirements

Update:
- `docs/claude-code-backend.md`

Add documentation for:
- structured continuation model
- continuity window policy
- warning carry-forward rules
- forwarding continuity behavior
- preserved limitations

## 12. Testing requirements

## 12.1 Unit tests
Add tests for:
- continuation context construction
- deterministic section rendering
- omission of empty sections
- warning relevance filtering
- forwarding continuity summarization
- truncation behavior under configured limits

## 12.2 Contract tests
Extend backend contract tests to validate:
- capability flags for structured continuation
- acceptance of continuation policy inputs
- stable rendered prompt semantics for identical state

## 12.3 Regression tests
Preserve:
- top-level MCP tool behavior
- API backend semantics
- warning normalization
- federation visibility constraints
- v0.6 forwarding guarantees

---

## 13. Acceptance criteria

v0.7.0 is complete when:

1. Claude Code continuation uses a structured continuation context
2. continuation prompt rendering is deterministic and test-backed
3. warning carry-forward is filtered by relevance
4. forwarding continuity is summarized, not raw repeated
5. session events expose continuity reconstruction decisions
6. top-level MCP contracts remain unchanged
7. docs describe new continuity behavior and preserved limitations honestly
8. full test suite passes

---

## 14. Explicitly preserved limitations

The following remain out of scope after v0.7.0:
- no native tool execution loop in Claude Code mode
- no true persistent Claude-side multi-turn session
- no streaming transport
- no cross-backend session migration
- no profile expansion track in this release

---

## 15. Recommended implementation phases

### Phase 1 — continuity models
- add continuation models
- add config fields
- add capability flags

### Phase 2 — reconstruction pipeline
- build structured continuation context
- implement bounded selection/truncation policy
- classify warnings and forwarding memory

### Phase 3 — prompt rendering
- render continuation-specific structured sections
- enforce deterministic ordering and omission rules

### Phase 4 — observability
- add continuation session events
- expose reconstruction stats

### Phase 5 — validation and docs
- complete tests
- update capability matrix and backend docs

---

## 16. Release note intent

v0.7.0 should be presented as:
- a continuity and ergonomics release for Claude Code mode
- a structured session reconstruction improvement
- not a claim of native API-equivalent session execution

---

## 17. Final recommendation

Implement v0.7.0 as a bounded continuity layer on top of the existing Claude Code backend.

Do not dilute the release with unrelated profile, transport, or federation expansion work.

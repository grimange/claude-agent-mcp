# claude-agent-mcp v0.5 — Claude Code Stabilization Track Architecture Specification

## 1. Purpose

This document defines the v0.5 **Claude Code stabilization track** for `claude-agent-mcp`.

The purpose of v0.5 is to make the `claude_code` execution backend materially more reliable, predictable, and usable in day-to-day operator workflows without changing the top-level MCP surface or redesigning the broader architecture.

v0.5 is not primarily a feature-expansion release.
It is a backend-quality release focused on improving the current Claude Code execution path introduced in v0.4.

---

## 2. Baseline assumed from v0.4

The stabilization track builds on the existing v0.4 baseline:

- explicit execution backend selection
- `api` backend
- `claude_code` backend
- canonical normalized MCP response envelopes
- durable SQLite-backed sessions
- local artifact persistence
- built-in profiles:
  - `general`
  - `verification`
- tools:
  - `agent_run_task`
  - `agent_continue_session`
  - `agent_get_session`
  - `agent_list_sessions`
  - `agent_verify_task`
- existing deployment transports
- existing session locking and crash recovery
- if v0.3 exists, downstream federation controls outside the Claude Code backend boundary

v0.5 must preserve these foundations.

---

## 3. Strategic objective

The stabilization track should answer this question:

**Can the Claude Code backend become reliable enough that operators can intentionally choose it for real work without confusion about what it can and cannot do?**

Success means the answer is yes.

---

## 4. Stabilization goals

v0.5 exists to deliver five improvements:

1. **Clear backend capability boundaries**
   The Claude Code backend should explicitly report supported and unsupported capabilities.

2. **More reliable continuation behavior**
   Continuation quality and predictability should improve without redesigning canonical internal sessions.

3. **More structured prompt/history reconstruction**
   Conversation history fed to the Claude Code backend should be more deliberate and less lossy than raw plain-text concatenation where practical.

4. **Better normalized output handling**
   Claude Code output should map more consistently into `claude-agent-mcp`’s normalized internal result model.

5. **Better operator documentation and diagnostics**
   Operators should understand setup, limitations, warnings, and failure modes clearly.

---

## 5. Non-goals

The following are out of scope for v0.5 stabilization track:

- new top-level MCP tools
- new built-in profiles
- replacing the current session model
- replacing SQLite
- removing the API backend
- making Claude Code backend fully feature-equivalent to the API backend
- adding unrestricted downstream federation forwarding into Claude Code backend
- redesigning the whole runtime around Claude Code-native sessions
- hosted multi-tenant architecture
- broad new transport work

This is a stabilization release, not a full parity release.

---

## 6. Current known limitations to address

From the v0.4 state, the Claude Code backend is known to have these limitations:

- it does not forward downstream federation tools
- it is effectively single-shot
- conversation history is serialized as plain text in the CLI prompt
- `stop_reason` is always normalized to `end_turn`

v0.5 should improve the second and third items directly, clarify the first item explicitly, and tighten normalization around the fourth item where feasible.

---

## 7. Core stabilization principles

### 7.1 Canonical internal sessions remain the source of truth

The Claude Code backend may improve how it reconstructs context, but internal session state remains canonical.

### 7.2 Honest capability reporting over false parity

If Claude Code mode still cannot do something the API backend can do, that difference should be surfaced clearly in docs and, where appropriate, runtime warnings.

### 7.3 Structured reconstruction over naive concatenation

Continuation context should be reconstructed more intentionally than a flat plain-text dump whenever practical.

### 7.4 Bounded improvements over speculative redesign

The goal is to harden the current backend path, not to invent a brand-new Claude Code orchestration system.

### 7.5 No silent semantic drift

Any changes to Claude Code backend behavior must keep top-level MCP contracts stable and documented.

---

## 8. Recommended work areas

v0.5 should focus on four technical work areas:

1. backend capability model
2. continuation/context reconstruction
3. output normalization
4. operator visibility and docs

---

## 9. Backend capability model

### 9.1 Purpose

The runtime should have an internal way to represent backend capabilities.

This lets the system reason clearly about what `claude_code` can and cannot do.

### 9.2 Suggested internal capability fields

Examples:

- `supports_downstream_tools`
- `supports_structured_tool_use`
- `supports_native_multiturn`
- `supports_rich_stop_reason`
- `supports_structured_messages`
- `supports_workspace_assumptions`

This can be a simple dataclass or lightweight model.

### 9.3 Behavior

The Claude Code backend should declare its capability set explicitly.
The workflow executor may use this to:

- add warnings
- adjust behavior safely
- prevent unsupported forwarding paths
- improve observability

This capability model should remain internal unless surfaced in sanitized docs or warnings.

---

## 10. Continuation and context reconstruction

### 10.1 Goal

Improve continuation quality while preserving the internal canonical session model.

### 10.2 Recommended approach

Replace naive flat plain-text history serialization with a structured reconstruction strategy.

Possible approach:

- summarize prior session state deliberately
- represent prior user/assistant exchanges with role markers
- include profile and task context separately
- include bounded recent turns in a consistent format
- include system-level instructions separately from prior conversational turns

Example conceptual prompt structure:

1. active system/profile instructions
2. current session summary
3. recent conversation history with clear role labels
4. current user continuation message
5. backend-specific execution instructions

### 10.3 Scope boundary

Do not redesign around native Claude Code session persistence as the canonical model.
If Claude Code-native continuation features are usable later, they may be optional optimizations, not the source of truth.

### 10.4 Truncation policy

If history is bounded, truncation should be explicit and deterministic.

Examples:
- most recent N exchanges
- bounded character/token budget
- include generated session summary when truncating

The policy should be tested and documented.

---

## 11. Output normalization

### 11.1 Goal

Improve consistency of Claude Code backend output mapping into `NormalizedProviderResult`.

### 11.2 Normalization requirements

The backend should normalize:

- main response text
- warnings
- detected partial failures
- backend capability caveats
- best-effort stop reason semantics
- backend metadata useful for observability

### 11.3 Stop-reason handling

If Claude Code CLI does not provide rich stop reasons, normalization should remain honest.

Possible options:

- keep `end_turn` as default but attach metadata or warnings
- introduce an internal “unknown” or “backend_defaulted” stop-reason marker before top-level normalization, if compatible with current types

Do not fabricate precision that the backend does not provide.

### 11.4 Verification robustness

If the verification workflow depends on structured backend text, make sure Claude Code output formatting is bounded enough to preserve fail-closed behavior.

---

## 12. Warning and diagnostics model

### 12.1 Runtime warnings

The Claude Code backend should be able to emit explicit warnings for conditions such as:

- downstream tools would have been visible but are not supported in this backend
- continuation context was truncated
- structured message fidelity was reduced for backend compatibility
- stop-reason precision is limited in this backend

### 12.2 Warning placement

Warnings should appear in normalized response fields where appropriate and/or in session events.
They should not silently disappear.

### 12.3 Operator-facing diagnostics

Diagnostics should make it easy to distinguish:

- missing Claude Code installation
- missing Claude Code login/auth state
- prompt reconstruction issues
- invocation timeout/failure
- unsupported capability conditions

---

## 13. Runtime integration

### 13.1 Workflow executor

The workflow executor remains the central orchestrator.

Its responsibilities continue to include:

- resolve profile
- consult policy engine
- resolve visible tools
- choose execution backend
- invoke backend
- normalize result
- persist session updates
- emit artifacts and warnings

### 13.2 Backend boundary

The Claude Code backend remains a backend implementation only.
It should not own session truth, federation policy, or top-level workflow semantics.

### 13.3 Capability-aware execution

The workflow executor may become capability-aware in limited ways, such as:

- surfacing warnings when backend capabilities do not match requested runtime affordances
- suppressing unsupported downstream tool forwarding safely
- choosing structured prompt reconstruction paths appropriate to the backend

---

## 14. Session metadata and observability

### 14.1 Metadata

Sessions may continue to record backend metadata such as:

- `execution_backend`
- backend-specific run identifiers if available
- warning counts or capability caveats if helpful

### 14.2 Session events

Session events should record relevant Claude Code backend observations such as:

- context reconstruction/truncation decisions
- unsupported-tool warnings
- backend invocation failures
- normalization caveats

This improves debugging without replacing canonical state.

---

## 15. Documentation requirements

v0.5 should add or improve docs for:

- Claude Code backend capability matrix
- setup and validation steps
- known differences vs API backend
- continuation semantics in Claude Code mode
- warning meanings
- troubleshooting failed or degraded backend execution

Suggested docs:

- `docs/execution-backends.md` update
- `docs/claude-code-backend.md` update
- `docs/backend-capability-matrix.md`
- README backend comparison section

---

## 16. Testing strategy

Required test categories:

1. **Capability declaration tests**
   - backend capability flags are explicit and stable

2. **Continuation reconstruction tests**
   - history formatting is deterministic
   - truncation is deterministic
   - role boundaries are preserved in generated prompt structure

3. **Normalization tests**
   - Claude Code output maps to normalized result fields correctly
   - warnings are surfaced consistently
   - stop-reason handling remains honest and stable

4. **Verification tests**
   - fail-closed behavior remains intact in Claude Code mode where applicable
   - malformed or underspecified output does not accidentally pass

5. **Diagnostic tests**
   - unsupported capability conditions emit warnings
   - missing prerequisite failures remain clear

6. **Cross-backend stability tests**
   - top-level MCP envelopes remain unchanged
   - API backend behavior does not regress

Do not rely on a live Claude subscription or external CLI login state in the default automated test suite.
Prefer test doubles and mocked subprocess behavior.

---

## 17. Recommended implementation phases

### Phase 0 — Scope freeze

Confirm:
- stabilization only
- no new top-level tools
- no broad feature parity promises
- no session redesign

### Phase 1 — Capability model

Add backend capability representation and surface it internally.

Checkpoint:
- Claude Code backend capabilities are explicit
- unsupported paths can be warned about clearly

### Phase 2 — Context reconstruction improvements

Refactor Claude Code prompt/history construction into a clearer structured builder.

Checkpoint:
- continuation reconstruction is deterministic and tested

### Phase 3 — Normalization and warnings

Improve Claude Code result normalization and warning surfacing.

Checkpoint:
- backend caveats appear clearly in normalized responses/events

### Phase 4 — Docs and capability matrix

Update docs with honest backend comparisons and usage guidance.

Checkpoint:
- operators can understand backend tradeoffs without reading code

### Phase 5 — Stabilization validation

Run full tests and summarize remaining backend differences honestly.

---

## 18. Minimum viable v0.5 stabilization release definition

A release qualifies as v0.5 Claude Code stabilization track only if all of the following are true:

- Claude Code backend capabilities are explicitly modeled or documented
- continuation/context reconstruction is more structured and deterministic than v0.4
- normalization warnings are surfaced clearly
- known backend differences vs API mode are documented clearly
- top-level MCP tool contracts remain unchanged
- API backend behavior does not regress
- tests cover capability, reconstruction, normalization, and diagnostics behavior

---

## 19. Recommended first slice

The best first slice is intentionally narrow:

- add capability model
- improve history reconstruction into a structured prompt builder
- add deterministic truncation rules
- surface unsupported-tool and limited-stop-reason warnings
- publish capability matrix docs

This is enough to materially improve operator trust without overcommitting to feature parity.

---

## 20. Summary

v0.5 Claude Code stabilization track is the right next step after v0.4 because the product now has dual backend support, but the Claude Code backend still needs clearer guarantees and better usability.

The correct next move is not broad expansion.
It is to make Claude Code mode:

- more predictable
- more diagnosable
- more clearly documented
- more trustworthy for intentional operator use

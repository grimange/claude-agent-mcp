# claude-agent-mcp v1.1.0 Architecture Spec
## APNTalk Restricted Verification Mode

## 1. Version

- Target version: `v1.1.0`
- Codename: `APNTalk verification mode`

---

## 2. Objective

Add a true restricted runtime mode for APNTalk that is:

- verification-only
- advisory-only
- bounded
- machine-verifiable
- fail-closed

This mode must run only on the `claude_code` backend and expose only the admitted MCP tool pair:

- `agent_get_runtime_status`
- `agent_verify_task`

The server itself must enforce this restriction at startup and at MCP tool registration time.

---

## 3. Why v1.1.0 exists

APNTalk does not trust downstream filtering or documentation-only restrictions.

It requires:

1. a server mode that actually publishes only the admitted tools
2. runtime-status proof that the exact restricted contract is active
3. fail-closed behavior if that contract cannot be satisfied

The current `claude-agent-mcp` runtime is broader than APNTalk’s admitted boundary, so a dedicated restricted mode is required.

---

## 4. Design goals

### Primary goals
- Add a first-class APNTalk verification mode
- Restrict the actual published MCP tool surface
- Make the restriction machine-verifiable through runtime status
- Fail closed if the restriction contract cannot be honored
- Keep implementation bounded and explicit

### Non-goals
- No API backend support for this mode
- No transport expansion
- No new execution features
- No broad policy framework redesign
- No hidden fallback to a broader runtime

---

## 5. Release theme

When started in APNTalk mode, the server becomes a verification-only MCP surface with exact runtime proof.

---

## 6. Architectural direction

Introduce a named restricted mode:

- `apntalk_verification`

This mode is hard-bound to:

- backend: `claude_code`
- transport: `stdio`
- authority posture: `advisory_only`
- tool surface: restricted
- exposed tools: exact admitted pair only

The mode should be activatable through at least one stable configuration path, preferably both:

- `CLAUDE_AGENT_MCP_MODE=apntalk_verification`
- `--mode apntalk_verification`

A generic allowlist may be added as an internal helper or future extension, but APNTalk must rely on the named mode contract, not ad hoc configuration.

---

## 7. Core capabilities

## 7.1 First-class restricted runtime mode

### Goal
Make APNTalk verification mode a real product mode, not a loose convention.

### Requirements
Add a runtime mode model or enum with at least:
- `standard`
- `apntalk_verification`

Add a resolved contract model, for example:
- `RuntimeRestrictionContract`

Suggested fields:
- `mode`
- `policy_mode`
- `authority_mode`
- `tool_surface_mode`
- `active_profile`
- `required_backend`
- `required_transport`
- `allowed_tools`
- `allowed_directories`
- `restriction_contract_id`
- `restriction_contract_version`
- `fail_closed`

### APNTalk resolved values
- `mode = apntalk_verification`
- `policy_mode = verification_only`
- `authority_mode = advisory_only`
- `tool_surface_mode = restricted`
- `active_profile = apntalk_verification`
- `required_backend = claude_code`
- `required_transport = stdio`
- `allowed_tools = [agent_get_runtime_status, agent_verify_task]`
- `restriction_contract_id = apntalk_verification_v1`
- `restriction_contract_version = 1`
- `fail_closed = true`

---

## 7.2 True server-side tool restriction

### Goal
Ensure MCP introspection shows only the admitted tools.

### Requirements
Tool restriction must happen at server construction / registration time.

When APNTalk mode is active, the server must publish only:
- `agent_get_runtime_status`
- `agent_verify_task`

The server must not advertise:
- `agent_run_task`
- `agent_continue_session`
- `agent_get_session`
- `agent_list_sessions`
- any broader execution-oriented tools beyond the admitted boundary

### Implementation direction
Use a mode-aware tool registration path:
- define full tool registry
- resolve mode contract
- register only tools allowed by the active contract

Do not register all tools and hide them later.

---

## 7.3 Exact runtime-status proof schema

### Goal
Allow APNTalk to verify the restricted runtime contract exactly.

### Requirements
Extend `agent_get_runtime_status` to include explicit restriction proof fields.

Suggested fields:
- `mode`
- `policy_mode`
- `authority_mode`
- `tool_surface_mode`
- `active_profile`
- `backend`
- `transport`
- `exposed_tools`
- `allowed_directories`
- `restriction_contract_id`
- `restriction_contract_version`
- `fail_closed_enabled`
- `restriction_compliance`
- `non_compliance_reasons`
- `server_version`

### Behavior
When APNTalk mode is active:
- proof fields must be present
- values must be exact and machine-readable
- `exposed_tools` must reflect the actual registered MCP surface
- `allowed_directories` must contain normalized exact paths

Do not return vague summary-only wording in place of proof fields.

---

## 7.4 Allowed-directory boundedness

### Goal
Make bounded filesystem scope provable.

### Requirements
In APNTalk mode:
- allowed directories must be explicit
- paths must be normalized to exact values
- paths must be provably bounded
- if repo-contained scope is required, validate that property at startup

### Behavior
Runtime status must expose the exact normalized allowed-directory list.

If bounded directories cannot be proven, fail closed.

---

## 7.5 Startup self-check and fail-closed behavior

### Goal
Prevent silent widening or partial compliance.

### Requirements
When APNTalk mode is requested, the server must validate at startup:

- backend resolves to `claude_code`
- transport resolves to `stdio`
- active profile resolves to `apntalk_verification`
- allowed tool set is exactly the admitted pair
- proof fields can be populated
- allowed directories are explicit and bounded
- no extra tools are registered

### Failure behavior
If any APNTalk contract requirement is not satisfied:
- startup should fail
- no broader fallback mode should be silently activated

Optional secondary behavior:
- if startup is intentionally allowed in diagnostic mode, `restriction_compliance=false` must be explicit

Recommended default:
- fail startup

---

## 7.6 Stable restriction identity

### Goal
Give APNTalk a stable contract identity it can verify.

### Requirements
Expose:
- `restriction_contract_id = apntalk_verification_v1`
- `restriction_contract_version = 1`

### Rationale
This allows APNTalk to recognize a known restricted contract instead of inferring it from loosely related fields.

---

## 7.7 Operator-visible startup reporting

### Goal
Make restricted-mode activation inspectable by operators.

### Requirements
At startup, log a concise resolved restriction summary including:
- mode
- backend
- transport
- active profile
- exposed tools
- allowed directories
- compliance / startup approval result

This is additive and should not replace machine-readable status output.

---

## 8. API and MCP contract impact

### Preserved behavior
No changes to the behavior of standard mode.

### Tool surface impact
In standard mode, existing tool surface remains unchanged.

In `apntalk_verification` mode, the published MCP surface is intentionally restricted to:
- `agent_get_runtime_status`
- `agent_verify_task`

This is not a breaking change to standard mode; it is mode-specific behavior.

---

## 9. Data model and config impact

### Suggested additions
- `RuntimeMode` enum or equivalent
- `RuntimeRestrictionContract`
- restriction proof fields on `RuntimeStatusSnapshot` or equivalent status model

### Config additions
At minimum:
- `CLAUDE_AGENT_MCP_MODE`
- optional CLI `--mode`

Optional later:
- generic tool allowlist config

For v1.1.0, the named APNTalk mode is the source of truth.

## 10. Documentation requirements

Update:
- `README.md`
- `docs/operator-guide.md`
- `docs/claude-code-backend.md`

Add or extend docs to cover:
- what APNTalk verification mode is
- how to enable it
- that it is `claude_code`-only
- exact exposed tool surface
- fail-closed behavior
- example runtime-status proof payload
- bounded directory requirements

Docs must clearly state that this mode is:
- verification-only
- advisory-only
- restricted
- machine-verifiable

---

## 11. Testing requirements

Add tests for:

### 11.1 Tool-surface restriction
- APNTalk mode publishes exactly:
  - `agent_get_runtime_status`
  - `agent_verify_task`

### 11.2 Forbidden tool absence
- `agent_run_task` absent
- session continuation tools absent
- session inspection tools absent unless explicitly admitted

### 11.3 Runtime proof exactness
- backend = `claude_code`
- transport = `stdio`
- active profile = `apntalk_verification`
- authority_mode = `advisory_only`
- tool_surface_mode = `restricted`
- exposed_tools exact match admitted pair
- allowed_directories exact normalized values
- restriction contract id/version present

### 11.4 Fail-closed behavior
- startup fails on backend mismatch
- startup fails on missing bounded directories
- startup fails if extra tools would be registered
- startup fails if proof fields cannot be constructed

### 11.5 Standard-mode regression safety
- existing standard-mode tool surface remains intact
- existing standard-mode tests continue to pass

### 11.6 MCP-surface level verification
At least one test should verify actual registered MCP tools, not only config resolution.

---

## 12. Acceptance criteria

v1.1.0 is complete when:

1. `apntalk_verification` mode exists as a first-class runtime mode
2. the server publishes only the admitted pair in that mode
3. runtime status proves exact restricted compliance fields
4. backend is hard-bound to `claude_code` in that mode
5. transport is hard-bound to `stdio` in that mode
6. bounded allowed directories are explicit and machine-readable
7. startup fails if the restriction contract cannot be honored
8. tests verify actual restricted MCP surface
9. standard mode behavior remains unchanged

---

## 13. Preserved limitations

Still out of scope for v1.1.0:
- no API-backend parity for APNTalk mode
- no transport expansion
- no new execution semantics
- no broad policy-engine redesign
- no hidden fallback to broader behavior

---

## 14. Recommended implementation phases

### Phase 1 — mode and contract model
- add runtime mode
- add restriction contract model
- add config/CLI activation path

### Phase 2 — tool registration restriction
- implement mode-aware MCP tool registration
- register only admitted tools in APNTalk mode

### Phase 3 — runtime proof schema
- extend runtime status with exact restriction proof fields
- expose normalized allowed directories and contract identity

### Phase 4 — startup self-check
- validate backend, transport, profile, tools, and bounded directories
- fail closed on non-compliance

### Phase 5 — tests and docs
- add MCP-surface restriction tests
- add runtime proof tests
- document APNTalk-safe startup path

---

## 15. Final recommendation

Implement v1.1.0 as a narrow, explicit APNTalk verification mode on top of the existing `claude_code` runtime.

Do not broaden scope into general policy redesign or multi-backend parity.

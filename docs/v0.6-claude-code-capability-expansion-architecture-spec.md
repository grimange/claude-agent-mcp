# claude-agent-mcp v0.6 — Claude Code Capability-Expansion Track Architecture Specification

## 1. Purpose

This document defines the v0.6 **Claude Code capability-expansion track** for `claude-agent-mcp`.

The purpose of v0.6 is to expand the practical usefulness of the `claude_code` execution backend after the v0.5 stabilization work, while preserving the explicit capability model, canonical internal session model, policy boundaries, and top-level MCP contracts.

v0.6 is a **bounded capability-expansion release**.
It is not a promise of full parity with the API backend.
It is not a redesign around Claude Code-native sessions.

---

## 2. Baseline assumed from v0.5

The capability-expansion track builds on the existing v0.5 baseline:

- explicit execution backend selection
- `api` backend
- `claude_code` backend
- backend capability model
- canonical normalized MCP response envelopes
- durable SQLite-backed sessions
- local artifact persistence
- built-in profiles:
  - `general`
  - `verification`
- tools:
  - `agent_run_task`
  - `agent_continue_session`
  - `agent_get_session`
  - `agent_list_sessions`
  - `agent_verify_task`
- existing deployment transports
- existing session locking and crash recovery
- existing backend capability warnings and docs
- if v0.3 exists, downstream federation controls outside backend ownership

v0.6 must preserve these foundations.

---

## 3. Strategic objective

The capability-expansion track should answer this question:

**Can the Claude Code backend become meaningfully more useful for real operator workflows without losing the clarity, safety, and boundedness achieved in v0.5?**

Success means the answer is yes.

---

## 4. Capability-expansion goals

v0.6 exists to deliver four bounded improvements:

1. **Selective downstream tool forwarding feasibility**
   Support a narrow, explicit, test-backed path for forwarding selected downstream tools into Claude Code mode, if practical.

2. **Better multi-turn / continuation usefulness**
   Improve continuation handling beyond structured prompt reconstruction, without abandoning the canonical internal session model.

3. **Clear capability downgrades and fallback warnings**
   When a requested capability cannot be realized in Claude Code mode, the system should degrade explicitly and transparently.

4. **Cross-backend contract confidence**
   Increase test coverage for equivalent tasks across `api` and `claude_code` where the contract should remain aligned.

---

## 5. Non-goals

The following remain out of scope for v0.6 capability-expansion track:

- claiming full feature parity between backends
- redesigning canonical internal sessions around Claude Code-native sessions
- unrestricted forwarding of all downstream tools
- dynamic downstream server registration
- new top-level MCP tools
- new built-in profiles
- replacing SQLite
- replacing the canonical response envelope
- removing explicit backend capability reporting
- removing or weakening policy/federation controls

This is an expansion of bounded usefulness, not a backend-unification release.

---

## 6. Current opportunity after v0.5

The v0.5 stabilization work improved:

- explicit backend capability modeling
- structured prompt reconstruction
- deterministic truncation
- clearer warnings
- more honest normalization

That means the next safe step is not more documentation alone.
The next safe step is to expand Claude Code mode in one or two carefully chosen areas while preserving the explicit capability truth model.

---

## 7. Core principles

### 7.1 Capability truth remains primary

Claude Code mode must continue to declare what it can and cannot do explicitly.

### 7.2 Expansion must be opt-in where risky

If a capability introduces meaningful uncertainty or trust-boundary expansion, it should be operator-controlled and disabled by default until proven stable.

### 7.3 Canonical internal sessions remain the source of truth

Even if continuation quality improves or downstream tools become partially usable, `claude-agent-mcp` internal state remains canonical.

### 7.4 No silent pseudo-parity

If Claude Code mode still differs from API mode, those differences must remain visible in warnings, docs, and capability reporting.

### 7.5 Smallest useful slice first

The first v0.6 implementation should expand one narrow useful capability rather than attempting broad parity.

---

## 8. Recommended v0.6 scope

The best v0.6 scope is:

1. **experimental selective downstream tool forwarding for Claude Code mode**
2. **continuation quality improvements**
3. **cross-backend equivalence tests for supported cases**

If downstream tool forwarding proves too awkward or brittle for Claude Code mode, the release may narrow to continuation improvements plus stronger capability downgrade behavior.

---

## 9. Selective downstream tool forwarding

## 9.1 Goal

Allow a small subset of already-visible, already-allowlisted downstream tools to be usable in `claude_code` mode when practical.

## 9.2 Scope restrictions

This must be narrow:

- disabled by default unless explicitly enabled
- only applies to tools already allowed by federation policy
- only applies to profiles that already permit those tools
- exact-tool forwarding only
- no wildcard mass forwarding
- no change to `verification` default posture unless explicitly configured and strongly justified

## 9.3 Forwarding model

Recommended first implementation:

- Claude Code backend receives a filtered set of visible tools
- only tools compatible with the backend’s forwarding mechanism are retained
- incompatible tools are dropped with explicit warnings
- forwarding decisions are logged in session events

## 9.4 Practical compatibility screen

The backend should screen tools for compatibility, such as:

- schema complexity
- argument-shape suitability
- tool-call invocation model
- result-size suitability
- safety classification if relevant

Tools that fail the compatibility screen should not be forwarded.

## 9.5 Capability flag

Add or refine a capability such as:

- `supports_limited_downstream_tools`

This is distinct from broad full downstream-tool support.

## 9.6 Warning model

If visible tools are filtered for compatibility, the response/session should indicate:

- which tool categories were not forwarded
- why they were not forwarded
- whether switching to the `api` backend would provide fuller support

---

## 10. Continuation improvements

## 10.1 Goal

Make continuation in Claude Code mode more useful and less lossy than the current v0.5 prompt reconstruction path.

## 10.2 Recommended improvements

Possible bounded improvements include:

- more selective inclusion of prior turns
- better session summary integration
- clearer separation between durable task context and conversational history
- specialized continuation prompt shape distinct from initial task prompt
- explicit carry-forward of unresolved objectives or pending work items when present

## 10.3 Scope boundary

Do not redesign around backend-native continuation semantics as the canonical model.
Native continuation features, if used at all, should be bounded optimizations.

## 10.4 Deterministic behavior

Continuation reconstruction must remain deterministic and testable.

---

## 11. Capability downgrade handling

## 11.1 Purpose

As Claude Code mode becomes somewhat more capable, downgrade handling becomes more important.

## 11.2 Required behavior

If the requested execution context implies capabilities Claude Code mode cannot fully support, the runtime should:

- attempt supported subsets where safe
- emit explicit warnings for dropped capabilities
- preserve top-level contract stability
- avoid pretending a capability was honored when it was not

## 11.3 Examples

Examples of downgrades:

- only some visible downstream tools forwarded
- rich stop-reason not available
- structured tool use approximated or unsupported
- continuation context truncated
- multi-turn assumptions reduced to single-shot bounded execution

---

## 12. Cross-backend contract confidence

## 12.1 Goal

Strengthen confidence that supported task classes behave compatibly across backends at the MCP-contract level.

## 12.2 Test focus

Add tests for cases where both backends should preserve:

- same top-level response envelope
- same error-shape conventions
- same session metadata invariants
- same profile/policy enforcement order
- same fail-closed verification behavior where feasible

Do not require identical underlying text outputs.
Focus on contract equivalence, not exact language matching.

---

## 13. Runtime integration

## 13.1 Workflow executor

The workflow executor remains the orchestrator.

Its responsibilities continue to include:

- resolve profile
- consult policy engine
- resolve visible tools
- reason about backend capabilities
- choose execution backend
- invoke backend
- normalize result
- persist session updates
- emit warnings and session events

## 13.2 Capability-aware tool forwarding

The workflow executor and/or backend may jointly perform:

- capability-aware filtering of visible tools
- downgrade warning generation
- explicit non-forwarding of unsupported tools

The backend must not bypass federation allowlists or profile visibility.

---

## 14. Configuration model

v0.6 may add narrow config options such as:

- `claude_code_enable_limited_tool_forwarding`
- `claude_code_tool_forwarding_allowlist` (optional if aligned with existing visibility)
- `claude_code_continuation_mode`

These should remain optional, explicit, and bounded.

Do not add a large new config surface unless necessary.

---

## 15. Session metadata and observability

## 15.1 Session metadata

Sessions may record additional bounded metadata such as:

- forwarded_tool_count
- dropped_tool_count
- continuation_reconstruction_mode
- capability_downgrades_present

## 15.2 Session events

Session events should record:

- which tools were considered for forwarding
- which were forwarded
- which were dropped and why
- continuation reconstruction decisions
- downgrade warnings emitted

This improves operator trust and debugging.

---

## 16. Documentation requirements

v0.6 should add or improve docs for:

- limited downstream tool forwarding in Claude Code mode
- capability downgrade examples
- continuation behavior improvements
- backend comparison updates
- when to prefer `api` vs `claude_code`

Suggested docs:

- `docs/claude-code-backend.md` update
- `docs/backend-capability-matrix.md` update
- `docs/claude-code-tool-forwarding.md` if needed
- README backend comparison update

---

## 17. Testing strategy

Required test categories:

1. **Tool-forwarding capability tests**
   - forwarding disabled by default if that is the chosen design
   - compatible tools can be forwarded when enabled
   - incompatible tools are filtered with warnings
   - `verification` remains conservative by default

2. **Continuation tests**
   - continuation builder remains deterministic
   - improved continuation modes preserve ordering and boundedness
   - unresolved-context carry-forward works if implemented

3. **Downgrade-warning tests**
   - partial forwarding emits clear warnings
   - unsupported capabilities emit clear warnings
   - warnings remain visible in normalized responses/events

4. **Cross-backend contract tests**
   - same top-level MCP envelope across backends
   - same structured error behavior where applicable
   - same policy/federation enforcement order

5. **Regression tests**
   - API backend behavior does not regress
   - v0.5 stabilization guarantees remain intact

Prefer test doubles for Claude Code execution and downstream tools.
Avoid live external dependency requirements in the default suite.

---

## 18. Recommended implementation phases

### Phase 0 — Scope freeze

Confirm:
- bounded capability expansion only
- no broad parity promises
- no top-level tool additions
- no session redesign

### Phase 1 — Forwarding feasibility layer

Add backend-side or runtime-side compatibility screening for limited downstream tool forwarding.

Checkpoint:
- unsupported tools are filtered safely
- forwarding remains opt-in if risky

### Phase 2 — Continuation improvements

Refine continuation reconstruction for Claude Code mode.

Checkpoint:
- continuation is more useful but still deterministic and bounded

### Phase 3 — Downgrade handling and observability

Surface downgrade warnings and record forwarding/reconstruction decisions.

Checkpoint:
- operators can see what was supported vs downgraded

### Phase 4 — Docs and tests

Update docs and add cross-backend/capability tests.

Checkpoint:
- operators can understand the new capability boundary clearly

### Phase 5 — Stabilization validation

Run tests and summarize remaining backend gaps honestly.

---

## 19. Minimum viable v0.6 release definition

A release qualifies as v0.6 Claude Code capability-expansion track only if all of the following are true:

- at least one meaningful Claude Code capability is expanded in a bounded, test-backed way
- capability downgrades are still explicit and visible
- top-level MCP tool contracts remain unchanged
- federation and policy controls are not bypassed
- API backend behavior does not regress
- tests cover forwarding/downgrade/continuation behavior where implemented
- docs explain the expanded capability and remaining limitations honestly

---

## 20. Recommended first slice

The best first slice is:

- add experimental limited downstream tool forwarding for compatible visible tools
- keep it disabled by default unless explicitly enabled
- improve continuation reconstruction modestly
- emit explicit downgrade warnings
- strengthen cross-backend contract tests

If forwarding proves too awkward, fallback to continuation-plus-downgrade improvements only.

---

## 21. Summary

v0.6 Claude Code capability-expansion track is the right next step after v0.5 because the backend is now honest and stable enough to justify cautious expansion.

The correct next move is not pretending to reach parity.
It is to make Claude Code mode:

- more useful
- still explicit
- still bounded
- still policy-aligned
- still understandable to operators
